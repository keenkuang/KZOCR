## C1 —— VLM 跨页内容泄漏 4 层防御系统

### L1: Prompt 约束 + 动态字符基线

| 项目 | 判定 |
|------|------|
| 开销 | 前 50 页字符数统计，O(n) 对已驻留内存字符串做 len() + 取中位数 |
| 计算 | 50 次 `len()` + 排序/中位数 ≈ **<0.1ms** |
| 运行时 | 仅在启动阶段一次，非逐页重复计算 |

- [x] **通过** — 开销可忽略（<0.1ms），不进入关键路径。

### L2: max_tokens=2048 物理上限（从 4096 下调）

| 项目 | 判定 |
|------|------|
| 开销 | 减少模型输出配额 |
| 计算 | 4096→2048 tok = 减少 50% 最大生成量 |
| 影响 | 因模型在输出接近上限前就会停止（中医书页平均 300-600 字 ≈ 450-900 tok），**实际很少触 2048 上限** |

- [x] **通过** — 实际性能影响近乎为零。4K→2K 的收紧主要是安全阀而非运行热路径。

### L3: 超阈自动重 OCR（char_count > baseline×1.5 → 用更紧 max_tokens 重 OCR，最多 1 次）

| 项目 | 判定 |
|------|------|
| 单次开销 | 多一次完整 API 调用 ≈ **+10s/页** |
| 触发概率 | baseline = 前 50 页中位数；threshold = baseline × 1.5。**正常书页字符数波动通常 <±30%**，典型触发率预期 **5-10%** |
| 平均摊还 | P(触发) × 10s / 页 ≈ 0.5-1.0s/页 额外开销 |
| 最坏情景 | 每页都触发 → 2× API 调用 = **+100% 时延**（~20s/页）|

**数据估算（50 页默认 / 500 页全书）：**

| 场景 | 无 L3 | 5% 触发 | 10% 触发 | 100% 触发（最坏） |
|------|-------|---------|----------|-----------------|
| 50 页 | 500s | 525s (+5%) | 550s (+10%) | 1000s (+100%) |
| 500 页 | 5000s | 5250s (+5%) | 5500s (+10%) | 10000s (+100%) |

- [x] **通过** — 典型场景 <10% 额外开销，在最坏假设下可超 20% 但天文概率。建议在监控指标中加入 `leakage_l3_retry_count` 以跟踪实际触发率。
- **建议**：若后续实测触发率 >15%，考虑提高 threshold 倍率至 1.8×，或加宽 baseline 窗口至前 80 页。

### L4: 增量探针重叠检测（合并阶段）

| 项目 | 判定 |
|------|------|
| 探针数量 | plen 从 50 到 300，step=10 → (300-50)/10 + 1 = **26 次迭代** |
| 单次开销 | Python `in` 子串搜索（C 层实现，Boyer-Moore 或改良算法） |
| 文本规模 | 单页中文 ~800 字 ≈ 2400 UTF-8 字节，norm_a 约 800 char |
| 总计算量 | 26 × 平均 200 char 探针 × 800 char 正文 ≈ 26 × O(800) ≈ **~20K char 操作** |
| 运行时机 | **仅当合并阶段检测到潜在泄漏时执行**，非每页必走 |

- **最坏情况搜索量**：
  - plen=50: 50×800 ≈ 40K 字符比较
  - plen=300: 300×800 ≈ 240K 字符比较
  - 总计梯度求和：~(50+300)×26/2 × 800 / 10 ≈ **364K 字符比较**
  - x86_64 CPU 上 Python C 层 strstr 约 2-4 GB/s → **<0.2ms**

- [x] **通过** — 开销可控在亚毫秒级，即使在最坏情景下也不进入性能关键路径。

### C1 总体结论

- [x] **通过** — 4 层防御的**总计性能开销 <5%（典型），最坏 <100%（仅 L3 全触发）但概率极低**。
- L1/L2/L4 开销均可忽略（综合 <1ms/页）；L3 是唯一有实际成本的层。
- L3 的设计（最多 1 次重试、超阈触发）保证了退化有界。
- 无任何层可在典型场景下导致 >20% 性能退化。

## C3 —— 自适应 3 层限流器

### Layer 1: AdaptiveRateLimiter（base_interval=6.0s）

**背景计算：** 当前单页 SenseNova 时延 **~10s**。

**6s 固定间隔的吞吐影响：**

| 维度 | 无限流 | 6s 间隔 | 退化率 |
|------|-------|---------|--------|
| 有效页速率 | 1/(10s) = 6 页/min | 1/(10+6)s = **3.75 页/min** | **-37.5%** |
| 50 页（默认） | 500s | 800s | **+60%** |
| 500 页（全书） | 5000s | 8000s (**2.2h**) | **+60%** |

- [ ] **需修订 — base_interval=6.0s 过于保守，与当前 ~10s 的单页时延不匹配**
  - 退避度计算：`overhead = interval / (page_latency + interval)` = 6/(10+6) = **37.5%** 纯等待开销
  - 注：这 37.5% 是 API 调用间硬闲置，不产生任何有用工作
  - 与 TOTAL_TIMEOUT=7200s 冲突：500 页 + 6s 间隔 = 8000s > 7200s，**书未跑完已超总预算**

**建议：**
- 将 base_interval 从 6.0s 降至 **2.0~3.0s**（退化率降至 ~17-23%）
- 或改为动态计算：`interval = page_latency × 0.3`，页速 10s 时 ≈ 3s
- 核心原则：interval 不应超过平均 API 时延的 **50%**，否则等待成本失控

### Layer 1: 503 间隔加倍到 60s

- 基准：base=6s → 12s → 24s → 48s → **60s** cap（约 4 次加倍后触及）
- 60s 上限：若 API 返回 503，等 60s 再试是非常合理的
- 与 `ExponentialBackoff`（max_delay=300s）配合形成双层退避

- [x] **通过** — 60s 上限合理。退避速度和 cap 对齐业界标准（AWS SDK 默认 max=60s）。

### TOC 管线 "4 节并行" vs v0.3 "MAX_CONCURRENCY=1"

**冲突分析：**

C5 TOC 管线 `ThreadPoolExecutor(max_workers=4)` 并行处理 4 节 OCR：
```
Worker 1: sec_A 页 → Worker 2: sec_B 页 → Worker 3: sec_C 页 → Worker 4: sec_D 页
```

而 v0.3 FREEZE §3 规定 `KZOCR_MAX_CONCURRENCY=1"（硬约束），v0.3 理由：
> 纯本地 VLM 时并发恒为 1，吞吐量受单 llama-server 实例上限约束

但 C3 限流器是给 **云端 SenseNova** 设计的，SenseNova 作为 SaaS API 天然支持并发请求（无单实例瓶颈）。"本地 VLM 并发=1" 的理由不适用于云端 API。

- [ ] **需修订 — 需显式区分云端 vs 本地并发约束**

**修正建议：**

设计层语意应改为：
| 引擎类型 | MAX_CONCURRENCY | 约束来源 |
|---------|----------------|---------|
| 本地 VLM (PaddleOCR-VL) | 1 | 单 llama-server 实例上限 |
| 云端视觉 (SenseNova) | ≤4* | 公司级配额 + 503 率决定 |
| 云端文本 (DeepSeek) | 1** | 500req/5h 硬限速 |

> *4 是 4 节并行所需；实际吞吐需看 Layer 2 公司级配额 (600 req/min) 是否够用
> **DeepSeek 后处理本身不构成 API 并发瓶颈（每节 1 次调用）

**4 节并行吞吐验证：**
- 4 个 SenseNova 并发 × ~10s/页 + adaptive interval (3s 建议值)：~13s/4页 = ~0.31 页/s
- 对比串行 ~10s/页 = 0.1 页/s
- 并行加速比 ≈ **3.1×**（理想 4×，因 interval 和上下文切换损耗）
- Layer 2 公司级配额 600 req/min：4 并发 × 60/13 ≈ 18.5 req/min，远未触及上限

- [ ] **需修订 — C3 应新增一条裁决："云端 SenseNova 在 Layer 1 限流保护下允许 max_concurrency=4"**

### Layer 2: MultiTokenRateLimiter（600 req/min，80% 触发等待）

- [x] **通过** — 阈值设计合理。4 节并行 + SenseNova ~10s/页 + interval ~3s = ~18 req/min，远低于 600 req/min 配额的 80% 告警线（480 req/min）。该层实际不构成约束，但作为安全阀保留。

### Layer 3: ExponentialBackoff（2s base, max=300s, 5 retries, jitter 0-50%）

- [x] **通过** — 标准退避设计，jitter 防惊群。Retry-After 尊重且 +1s 安全边际正确。

### C3 总体结论

| 子项 | 判定 |
|------|------|
| Layer 1 base_interval=6s | - [ ] **需修订** — 过于保守，→建议 2-3s |
| Layer 1 503 double→60s | - [x] 通过 |
| Layer 2 MultiToken 600 req/min | - [x] 通过 |
| Layer 3 ExponentialBackoff | - [x] 通过 |
| TOC 4 节并行 vs MAX_CONCURRENCY=1 | - [ ] **需修订** — 需区分云端/本地并发规则 |

## C2 —— 原子写入 + 文件存在性断点续传

### 批量写入性能分析

`atomic_write()` 实现：
```python
tmp = path.with_suffix(path.suffix + ".tmp")
tmp.write_text(content, encoding="utf-8")  # ①
os.replace(tmp, path)                       # ②
```

**关键观察：`Path.write_text()` 不调用 `fsync` / `flush()`**

| 操作 | 耗时估算 | 说明 |
|------|---------|------|
| ① `write_text()` | **~5-50µs** | 纯内存写 + page cache，无 fsync，无磁盘 IO |
| ② `os.replace()` | **~1-10µs** | 同文件系统 rename，元数据操作 |
| 合计/次 | **~10-60µs** | 纯 CPU 操作，不入存储 IO 路径 |

**批量估算（500 页全书）：**

| 文件类型 | 文件数/页 | 总文件数 | 总写入耗时 |
|---------|----------|---------|-----------|
| 页级 .md（VLM 路径） | 1 | 500 | **~5-30ms** |
| TOC 管线文件（raw/cleaned/formula/manifest） | 4-6/节 | ~200-300 | **~2-18ms** |
| 合计 | — | 700-800 | **~7-48ms** |

**崩溃安全与 fsync 取舍：**

当前实现**不做 fsync**，这意味着：
- 系统崩溃时最近写入可能丢失，但 `is_complete()` 检测到不完整文件会重新处理
- 已经 `os.replace()` 的旧版本是完整的（原子 rename）
- 这是合理的权衡：**写入速度优先，正确性由重处理保证**

- [x] **通过** — 对全书 500 页的总额外开销 <**50ms**，在 5000s 总处理时间里完全可忽略（<0.001%）。
- 设计决策正确：前端不引入 fsync，崩溃恢复由 `is_complete()` + 重处理保障。

### 文件存在性断点续传性能

- `is_complete()` 单次调用：`path.exists()` + `stat().st_size > 0` = **~2 次 syscall = ~1-5µs**
- 每节/每页启动时检查目标文件存在性
- 500 页重启时：至多 500 次检查 ≈ **~2.5ms**

- [x] **通过** — 可忽略。若多次重启（最坏每次中断都重跑全书检查）也只增加几毫秒。

### atomic_write → `os.replace()` 后端文件系统兼容性

- `os.replace()` 在以下文件系统上是原子的：**ext4、xfs、btrfs、zfs**（同文件系统 rename）
- 在 **NFS/CIFS** 上 rename 不保证原子性

- [x] **通过** — 当前部署环境（Linux ext4）行为正确。

### C2 总体结论

- [x] **通过** — 性能无虞。全书 500 页写入总开销 <**50ms**（占 5000s 总时间的 0.001%）。断点续传检查也是微秒级。

## C5 —— TOC 驱动管线补充设计

### 渐进扫描策略：scan_ranges = [[1,5], [1,10], [1,21]]

| 扫描轮次 | 页码范围 | SenseNova 调用次数 |
|---------|---------|------------------|
| 1 | 1-5 | 5 次（+ 5 页×15% 上下文） |
| 2 | 1-10 | 5 次增量（复用时标为已 OCR） |
| 3 | 1-21 | 11 次增量 |

- **最坏：3 轮共 21 次调用**，但前轮结果可缓存，实际增量调用约 21 次
- 21 × 10s = **210s（3.5min）** TOC 分析阶段
- 若按固定前 21 页一次扫描：21 × 10s = 同样 210s
- 渐进策略的实质收益：**早停**（若前 5 页已识别出足够结构，跳过后续轮次）

- [x] **通过** — 渐进策略在最坏情况下不退化为比固定扫描更慢。早停收益纯赚。

### 页码偏移处理（start_pg = book_start + page_offset）

- 仅为一个整数运算，可忽略

- [x] **通过**

### TOC 为权威边界

- **性能增益**：TOC 边界避免了 OCR 推断边界时的额外计算（不用试探法估算章节起止）
- 减少跨节合并的 O(n) 扫描范围

- [x] **通过** — 边界精确化实际减少了合并阶段的字符串搜索量，属于**净性能优化**。

### C5 总体结论

- [x] **通过** — TOC 管线其自身的性能开销可控（TOC 分析阶段 ~3.5min），且为全书处理提供了精确边界。
- 与 C3 的并发冲突已在 C3 评审中标记。

## 跨项交叉问题

### 跨项冲突 1（严重）：C3 6s interval + TOTAL_TIMEOUT 7200s 导致 500 页全书不可完成

```
无 interval: 500 页 × 10s = 5000s (1.4h) → 在 7200s 预算内
有 interval:  500 页 × 16s = 8000s (2.2h) → 超出 7200s 预算
```

- 根源：C3 AMEND 引入了新的 6s/page 等待开销，但未相应调整 TOTAL_TIMEOUT
- 影响：默认配置下全书 500 页触发总预算超时，产生大量 HumanGate
- [ ] **需修订** — 要么降低 interval（→建议 2-3s），要么提高 TOTAL_TIMEOUT（→建议 10800s）

### 跨项冲突 2（中）：TOC "4 节并行" × Layer 1 interval 的真实吞吐

4 节并行的有效间隔取决于 Layer 1 是**共享**还是**各实例独立**：

| 模式 | 行为 | 4 节吞吐 | 对比串行 |
|------|------|---------|---------|
| 共享限流器 | 4 个 waiter 竞争同一 `last_call` | ~0.24 页/s | +2.4× |
| 独立限流器 | 各实例有独立 `last_call`，起始错开 | ~0.31 页/s | +3.1× |

- [ ] **需修订** — 设计应明确限流器实例化策略。建议：
  - Layer 1 AdaptiveRateLimiter **每适配器实例独立**
  - Layer 2 MultiTokenRateLimiter **全局共享**（公司配额唯一）
  - TOC 管线每个 worker 持有自己的 AdaptiveRateLimiter 实例

### 跨项冲突 3（中）：C2 atomic_write × TOC 管线文件数

TOC 管线每节产生 4-6 个文件（raw.md, cleaned.md, formulas.json, manifest.json, formula_N.md...），50 节全书 → 200-300 文件写入。但如前分析，总写入成本 <50ms，不存在性能问题。

- [x] **无冲突** — 文件数量级合理，写入性能不构成瓶颈。
