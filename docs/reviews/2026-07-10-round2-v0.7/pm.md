## 新增产品关注点

### N1：人工校对反馈闭环缺少变更日志（中）

其他评审角色已确认 `review_manifest + feedback_apply()` 的设计完整。但从产品视角看：

- `feedback_apply()` 将人工校对结果回写到 `confusion_set.json`、`rare_allowlist.json`、`toxic_herbs.json`
- **但缺少变更日志** — 运维/领域专家无法追踪"谁在什么时候改了哪个资源文件"
- 建议仿照 git 提交日志格式，在资源文件目录下维护一个 `CHANGELOG-resources.md`，每次 `feedback_apply()` 追加一行：

```
2026-07-10T14:32:08 | 用户校验 金匮要略-卷上 | add_confusion: 己→已 | review_manifest-20260710-143208.json
```

**风险**：如果多用户同时校对并回写，缺少日志将导致异常回写不可追溯。

### N2：过渡期的用户沟通计划（低）

`use_vlm` / `vlm_engine` 等旧环境变量的废弃时间线在方案中缺失。对用户而言：

- v0.7 初始版本：旧式环境变量仍然有效（通过委派模式转换）
- 建议 v0.8 时：明确废弃，发布 changelog 告知用户迁移方式
- 实施时需在项目 README 中增加"v0.7 配置迁移指南"章节

### N3：迁移期文档的用户视角切换（低）

当前 README 的"快速开始"章节基于 v0.6 的 `KZOCR_USE_VLM=1` 模式。v0.7 上线后：

- 新用户首次接触时应当看到的是 `kzocr pipeline` + 调度器
- 旧用户需要知道从 v0.6 怎么平滑升级
- 建议在 README 中并行维护两套快速开始（v0.6 旧方式 / v0.7 调度器方式），v0.8 移除旧方式

## 版本对照：v0.6 用户 → v0.7 用户

第一轮我评测说"v0.7 解决的是工程问题而非用户问题"，修订版已证实这个判断需要修正——

**v0.7 最终交付物对用户的价值：**

| 用户体验维度 | v0.6 | v0.7 | 改善程度 |
|-------------|------|------|---------|
| OCR 质量 | 单引擎单次，无校验 | 三级兜底 + 字形验证 | ⬆ **显著提升** |
| 透明性 | 黑盒，不知用哪个引擎 | 引擎报告 + trace | ⬆ **从无到有** |
| 控制力 | 仅有环境变量开关 | `--engine` / `--prefer` / `--tier-limit` CLI 覆写 | ⬆ **大幅增强** |
| 速度感知 | 无声等待 | 进度条 + ETA | ⬆ **从无到有** |
| 失败恢复 | 全书失败或吞错误 | 逐页降级 + HumanGate | ⬆ **质变** |
| 调试能力 | 日志中有调用记录 | `benchmark status` + trace JSON | ⬆ **从弱到强** |
| 配置复杂度 | 多处环境变量 | 统一 CLI + Config 自动迁移 | ⬆ **简化** |

两轮评审间从"工程师写给工程师"进化为"可面向用户交付"的设计。GlyphVerifier + 三级兜底 + 引擎报告这三项组合在一起，对用户的 OCR 产出质量和可信任度都将产生可感知的提升。
