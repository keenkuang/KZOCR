## 总体判断

**有条件通过（CONDITIONAL PASS）**。修订版在第一轮指出的 6 项领域问题中，有 **3 项已充分解决**（竖排盲区、毒药剂量校验、人工校对反馈闭环），**1 项部分解决但需补充边界**（调度器领域感知），**2 项未解决**（GlyphVerifier 异体字覆盖面、资源使用）。此外第一轮提出的 3 项概念遗漏中有 2 项（眉批夹注、方剂表格）仍未被触及。

## 六项领域问题复查

### 1. 竖排盲区 — 第一轮 P0

**第一轮核心意见**：T1 引擎全不支持竖排，竖排古籍实际兜底链降为 T2→T3→Human，方案需在调度层做竖排感知。

**修订版处理**：

- **4.3 竖排感知调度**：`select_candidates()` 在 tier==1 且 `page_layout.is_vertical` 时直接 `return []`，跳过 T1 全体引擎。这是对第一轮 P0 的核心整改。
- **4.4 领域感知权重**：`domain_adjust()` 中竖排页对 T1 降权 0.3。需注意：4.3 的提前 `return []` 意味着 4.4 的降权分支实际上不会被执行（竖排页连候选集都不进）。建议将 4.3 的 `return []` 作为最终仲裁，4.4 的 `*0.3` 保留但不期望触发，两处一致即可。
- **5.4 设计决策**：明确竖排检测在 Scheduler 层而非 Verifier，理由充分（调度决策而非验证）。
- 架构图 2.1 标注了"竖排检测在 Scheduler 层，非 Verifier"。

**裁决：✅ 已解决。** 竖排感知调度已完整纳入设计，从架构层面做了正确放置（Scheduler 而非 Verifier），且实现了 Tier 1 跳过逻辑。建议在文档中补充 T1 引擎竖排能力的具体标注（如 paddleocr/mineru/rapidocr/unirec 的竖排支持现状），避免用户对三级设计的实际覆盖范围产生误解。

### 2. GlyphVerifier 异体字覆盖面 — 第一轮 P1

**第一轮核心意见**：
1. variant_map 仅 ~50 对，实际需求 >500 对
2. 缺少偏旁层级匹配（Unihan 部首分解）
3. 缺少 TCM 领域词频感知
4. 缺少上下文消歧（bigram/trigram）

**修订版处理**：

- **5.1 Detector 协议**：`VerifierContext` 增加了 `neighbor_texts` 和 `book_code`，为上下文消歧提供了数据传入通道。
- **5.2 预注册检测器列表**：维持 5 个检测器不变，未新增偏旁分解检测器或 TCM 频率检测器。
- **Section 9 关联文件**：variant_map.json、confusion_set.json 均标注为"无变更"。
- 整体未看到对 variant_map 规模的扩展承诺、偏旁层级匹配的实现、或 TCM 领域词频表的引入。

**裁决：❌ 未解决。** 这是第一轮最复杂的领域问题，修订版仅在数据通道层面提供了基础（`neighbor_texts`），但三项实质性缺陷均未被触及：

1. **variant_map 规模问题**（50→500+）：无扩展计划。冷启动期大量异体字会被标记为 UNKNOWN→T2/T3，导致不必要的 API 调用。
2. **偏旁层级匹配**：无设计。面对"艹-⺾混用、辶-⻌混用"等偏旁级别的变体，整字映射完全无能。
3. **TCM 领域词频感知**：无设计。"砒""瘛瘲"等 TCM 高频但通用罕见的字仍会被误判。

**建议严重程度降级为 P2 处理**：v0.7 可以接受 variant_map 保持现有规模 + 依赖 T2/T3 兜底 + HumanGate 兜人，作为最小可行版本。但需在文档中明确标注异体字覆盖的局限性，并作为 v0.8 或 v1.0 的可量化目标。

### 3. 毒药剂量校验 — 第一轮 P1

**第一轮核心意见**：`toxic_herbs.json` 已有 `max_dosage_g` 但未被 GlyphVerifier 使用，OCR 误读毒药剂量可造成临床风险。

**修订版处理**：

- **5.3 ToxinDoseDetector**：全新的检测器类，完整实现了：
  - 从 `toxic_herbs.json` 加载 `{herb_name: {max_dosage_g}}`
  - 正则匹配 `(药名) + (数字)g` 模式
  - 超限时返回 `GlyphStatus.FAIL` + `confidence=1.0`
  - `details` 携带 `herb`, `dosage`, `max`, `severity=critical`
- **5.2 检测器优先级**：ToxinDoseDetector 置于最高优先级（FAIL/CRITICAL 立即终止），逻辑正确——触及安全时不应有后续检测器覆写。
- **Section 9**：`toxic_herbs.json` 标注为"首次接入 GlyphVerifier"。

**裁决：✅ 已解决。** ToxinDoseDetector 的设计完整、定位正确、优先级合理。正则模式 `{herb}\s*(\d+)\s*g` 覆盖了常见的中药剂量表达。

**两点边界建议**（非阻塞）：
- 中医学中"钱"（1 钱 ≈ 3g）也是常见单位，但正则只匹配 `g`。建议扩展为 `(g|克|钱)` 并在汇总时统一转换为克。
- 部分 toxic_herbs 的剂量在不同炮制方法下上限不同（生附子 vs 制附子），当前 `HerbEntry` 结构是否支持？如不支持至少要在 `details` 中留下扩展余地。

### 4. 调度器领域感知 — 第一轮 P1/P2

**第一轮核心意见**：
1. 延迟权重惩罚 VLM，与"视觉 LLM 才是竖排古籍实际有效引擎"矛盾
2. "通过率"分母盲点（10 字全对 vs 1000 字错 20 个）
3. 无出版时代感知（雕版印刷本应优先选 VLM）

**修订版处理**：

- **4.4 `domain_adjust()`**：新增领域感知调整函数，涵盖：
  - 竖排×0.3（T1 降权）
  - 雕版印刷×(1.5)（T2/T3 提权）
  - 表格×0.5（T1 降权）
  - 冷启动×0.8（样本不足降权）
- **第一轮第 1 点（延迟惩罚 VLM）**：在竖排场景下，4.3 的跳过 T1 + 4.4 的雕版提权 1.5，实际上使 VLM 在竖排古籍场景下获得了优先调度。但第一轮提出的"延迟权重在通用排序公式中惩罚 VLM"问题——当前排序使用 `bayesian_score × decay` 而非 `pass_rate / latency`，贝叶斯平均已弱化延迟的单一影响，但延迟仍通过 `avg_latency` 间接影响 `bayesian_score`。VLM 的延迟通常比 OCR 高 10-50 倍，即使竖排页跳过了 T1，在 T2 内排序时高延迟仍可能使其排在候选末尾。**建议**：在 `domain_adjust()` 中对竖排/雕版场景增加 `has_priority` 字段，使符合条件的引擎在 tier 内排序时获额外提升。
- **第一轮第 2 点（通过率分母盲点）**：修订版的贝叶斯平均公式（`n = total_pages`, `C = 7`）部分缓解了该问题——处理页数不足引擎的通过率被先验拉回。但未引入"单页平均识别字数"作为辅助指标。**影响较低**，贝叶斯 + `total_pages < 10 → ×0.8` 的组合已足够在冷启动期提供保守估计。
- **第一轮第 3 点（出版时代感知）**：已在 `domain_adjust()` 中实现。

**裁决：🟡 部分解决。** `domain_adjust()` 的补充大幅改善了领域感知能力，贝叶斯平均也缓解了重点部分问题。但竖排/雕版场景下 VLM 的延迟惩罚未完全解决（T2 内部排序时高延迟引擎仍落后）。建议在竖排感知的调度流程中，增加一个"场景优先级"修饰：当 `page_layout.is_vertical == True` 且 `tier >= 2` 时，将 VLM/LLM 引擎的评分提高一个权重系数（如 +0.2 偏移量），确保它们不会被同 tier 的低延迟引擎排挤。

### 5. 资源使用 — 第一轮 P1

**第一轮核心意见**：
1. 缺少 TCM 字符频率表（用于 RARE/UNKNOWN 判断）
2. rare_allowlist 标记策略过保守（RARE→PASS）
3. confusion_set 应兼做纠错修正映射

**修订版处理**：

- **第 2 点（rare_allowlist PASS）**：Section 9 标注 `rare_allowlist.json` 的 TermKBMatcher 增强为"RARE→PASS"，第一轮已确认。**已解决。**
- **第 1 点（TCM 字符频率表）**：未引入。这导致 RARE/UNKNOWN 的判定缺少领域语义。"瘛瘲"在儿科书籍中每日出现，却被判定为 UNKNOWN→触发 T2/T3。
- **第 3 点（confusion_set 兼做纠错映射）**：未涉及。

**裁决：❌ 未解决。** TCM 字符频率表的缺失是领域资源层最突出的缺口。与 GlyphVerifier 覆盖面问题（第 2 项）相关联，两者叠加导致冷启动期 RARE/UNKNOWN 过多。

**建议**：
- 将 TCM 字符频率表列为 **v0.7 后 2 周内可追加的增强**，规模不大（从现有 500+ 已处理书籍语料中统计即可），但对验证器准确率提升显著。
- 也可考虑先将 rare_allowlist 规模从 ~22 项扩充至 100+ 项作为短期替代方案，再逐步过渡到频率表。

### 6. 人工校对反馈闭环 — 第一轮 P1

**第一轮核心意见**：
1. 不做优先级区分
2. 无结构化证据输出
3. 无反馈回路
4. human_final 字段未参与编排

**修订版处理**：

- **5.6 完整的数据结构**：`ReviewManifest`、`ReviewPageItem`、`ReviewIssue`，覆盖了第一轮建议的全部字段：
  - 优先级：P0=FAIL, P1=UNKNOWN, P2=RARE
  - 结构化证据：`engine_results: dict[str, str]`（每级引擎产出）、`crop_img_path`、`issues` 详细列表
  - `issue_type`: glyph/dosage/herb/layout
  - `severity`: critical/warning/info
- **`feedback_apply()` 函数**：定义了三条反馈路径：
  1. 确认的 glyph 纠错 → confusion_set.json 追加
  2. 确认的稀有术语 → rare_allowlist.json 追加
  3. 修正后的剂量值 → toxic_herbs.json 校验
- **Section 9**：`human_final` / `ProofreadRecord` → 标注为"新增：review_manifest + feedback_apply()"
- **Phase 3 实施**：3.8 明确列入了 review_manifest + feedback_apply，预计 1 天。

**裁决：✅ 已解决。** 从第一轮的"一行字符串"到完整的数据结构 + 回写机制，改进幅度大。结构和回写路径的设计合理，与现有 ProofreadRecord 的衔接明确。

**非阻塞建议**：
- `feedback_apply()` 建议增加**变更日志**（changelog），记录每次回写操作的时间、来源书籍、操作类型（`add_confusion_pair` / `add_rare_term` / `update_max_dosage`），便于事后审计和回滚异常回写。
- 在实施中注意 `feedback_apply()` 的幂等性和并发安全——多个 `review_manifest` 可能同时更新同一资源文件。

## 最终建议

### 实施前必须调整

N/A — 修订版已无 P0 阻塞性问题。

### 建议在 Phase 2 中等优先级处理

1. **`domain_adjust()` 增加"竖排/雕版场景优先级偏移"**：当 `page_layout.is_vertical == True` 且 `tier >= 2` 时，对 VLM/LLM 引擎的评分增加一个正向偏移（如 +0.2），确保它们在高延迟下仍被优先选中。这是针对第 4 项剩余问题的最小改动。

### 建议在 v0.7 之后、v1.0 之前处理

2. **TCM 字符频率表**：从已处理语料中统计字符频率（约 1-2 天工作量），接入 GlyphVerifier 的 RARE/UNKNOWN 判定。
3. **variant_map 扩展**：从 ~50 对扩展到至少 200 对（基于现有 corpus 的高频异体字统计）。
4. **偏旁层级匹配**：作为 GlyphVerifier 的回退检测器实现，优先级低但扩展性强。

### 不建议在 v0.7 范围内追加

5. 眉批/夹注/方剂表格专用路径：v0.8 再议。
