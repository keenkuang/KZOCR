## 逐项复核（确认 R6/R7/R8 结论的稳定性）

### D0 — Config 扩展（`kzocr_output_dir`）

rc4 新增 `_compute_config_hash()` 隐含使用了 `cfg.kzocr_output_dir` 路径。该路径已明确通过 C2 `_check_base` 校验（实施说明第 3 条）。

**R7 §4.1 `/tmp` 权限问题维持：** 实现时需 `vlm_cache/` 显式 `chmod(0o700)` + `atomic_write` 默认 umask 的关注。非 blocker。

**状态：** 维持通过。

### D1 — 异常分类 + `retry_with_policy`

rc4 新增 `RateLimitedError.__init__(retry_after)` 参数。这是纯构造函数参数扩展：
- `retry_after` 来源为 HTTP `Retry-After` header（由适配器层传递）
- 传递给 `retry_with_policy` 后仅用于退避延迟计算
- 不进入日志、不写入文件、不参与任何数据路径

**状态：** 维持通过。D1 异常消息泄漏面未扩大。

### D2 — VLM 主循环重试

rc3 的 `page_num` 闭包捕获修复 + rc4 的 `base_delay` 调优。rc4 无涉及 D2 逻辑的其他变更。

**状态：** 维持通过。失败分类记录（`failed_pages: dict[int, str]`）内容不变（页码 → 异常类型名），无敏感数据。

### D3 — VLM 缓存（关键防御链）

rc4 新增 `_compute_config_hash()` 完整实现定义。确认：
- 参与哈希的参数：engine、sensenova_model、sensenova_base_url、vlm_host、vlm_port — 均非密码/密钥/凭据
- 输出：SHA-256 摘要前 16 字符 — 仅用于缓存元数据匹配
- 不引入任何新的敏感数据持久化路径

R7 已验证的防御链完整且稳定：
