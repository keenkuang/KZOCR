## 1. 重试延迟（1s → 2s → 4s）用户体验评估

**结论：可接受，但需要用户可见性。**

### 单页表现

| 尝试 | 延迟（加抖动） | 用户感知 |
|------|---------------|---------|
| 第 1 次失败 → 等待 | ~0.5–1.5s | 几乎不可察觉 |
| 第 2 次失败 → 等待 | ~1.0–3.0s | 稍有停顿 |
| 第 3 次失败 → 跳过 | — | 页面失败已确认 |

单页 3 次重试累计约 2–8s，对于一次 OCR 会话而言，用户的耐心窗口在数秒量级，**不会引起挫败感**。

### 批量场景风险

假设一本 100 页书、10% 页面存在瞬时 API 故障：
- 10 页 × 平均 ~4s/页 = 额外 ~40s
- 体验尚可，但与进度反馈叠加后可能让用户误以为"卡住了"

对于 `RateLimitedError` 的 `Retry-After` 场景（可能长达 30s+），**文档已注明尊重 header**——这是 API 侧的要求，非我方可控。但需要向用户透明展示等待原因（见条件 A）。

### 性能验证（round8 采纳）

`base_delay` 从 2.0s 降为 1.0s 是正确决策——首轮重试 1s 的感知延迟远低于 2s，且在批量场景中累积节省显著（50 页 × 10% 故障 = 节省 ~5s）。确认采纳。

## 2. 缓存恢复功能（D3）的 P2 优先级评估

**结论：P2 合理。** 用户价值显著，但不应高于 D1+D2。

### 用户价值分析

| 场景 | 无缓存 | 有缓存 | 用户获益 |
|------|--------|--------|---------|
| 50 页方剂书，网络中断于第 40 页 | 重头开始（损失 100% 进度） | 从第 41 页继续（损失 0%） | **极高** |
| 参数微调后重跑 | 全量重 OCR | 仅未缓存页重 OCR | **高** |
| 长周期批量任务（200 页+） | 中断即前功尽弃 | 可逐步积累 | **高** |

### 为何 P2（非 P1）

- **缓存依赖 D2 的重试机制**：如果没有 D2 的 `retry_with_policy` 和 `failed_pages` 追踪，缓存在语义上不完整（缓存了"重试耗尽"的状态吗？）
- **D2 的价值面更广**：D2 影响每次运行的每个页面（所有用户受益），D3 仅影响中断恢复场景（子集受益）
- **但 D3 的边际成本低**：同一人、同一函数（`_run_vlm`），技术风险不大

> **建议评估**：如果 D2 实施后 `_run_vlm` 重构充分（提取 `_process_vlm_page`），D3 可以作为"把一个函数调用改成先读缓存"的微变更并入 P1，不增加工程风险。取决于团队的冲刺节奏。

### 策略建议

- **不主动推广 P2→P1**，但**允许工程师在 D2 实施后自然延展**。如果 D2 PR 未增加太多复杂度且 D3 改动幅度小，接受合并。
- 如果拆为独立 PR（D2 → D3），**D3 优先于其他 P2 任务**。

| 当前顺序 | 价值流向 | 评价 |

### 值得注意的空隙

- D1: "用户不再看到 `Exception: ...` 原始 Python traceback" → 验收标准
- D2: "90%+ 瞬时故障页面通过自动重试恢复" → 可观测
- D3: "中断后重跑不重复处理已完成页面" → 可验证

## 4. 缺失的用户面功能

计划在技术层面覆盖充分，但以下 3 项用户侧功能是缺失的：

### A. 重试过程中的用户反馈（建议：**必须**）

当前：VLM 主循环有 `logger.warning` 和 `logger.error`，但没有在 CLI 输出或进度条中反映重试状态。

用户看到："第 37 页处理中…"（10 秒后）"第 37 页处理中…"——用户不知道发生了什么。

**建议修改：**
```python
# 在 retry_with_policy 的 on_exhausted 之外，增加尝试间日志
# 第 1 次失败时：
logger.warning("[VLM] 第 37 页 API 失败（连接超时），1.2s 后重试（2/3）…")
# 第 2 次失败时：
logger.warning("[VLM] 第 37 页 API 失败（连接超时），2.5s 后重试（3/3）…")
# 耗尽时：
logger.error("[VLM] 第 37 页 3 次重试耗尽，跳过：API 连接超时")
```

这不是 `retry_with_policy` 的责任——它不知道上下文。**应由 D2 的 `_run_vlm` 或 `_process_vlm_page` 在调用 `retry_with_policy` 前/后包装日志。** 最简单的做法：`on_exhausted` 回调中加 filename 日志，但这不足以让用户感知到"正在进行第 2 次尝试"。

**实际实施：** 在 `_process_vlm_page` 中，在 try/except 块内捕获 `ApiError` 时主动输出带尝试次数的日志。

### B. 会话结束总结报告（建议：**必须**）

`failed_pages: dict[int, str]` 当前仅在内存中。用户不会去翻日志。

**建议：** `_run_vlm` 结束时输出一个总结行到 stdout：
```
[VLM] OCR 完成：47/50 页成功，3 页失败（P2-ApiError, P15-RateLimitedError, P37-ApiError）
```

或者扩展 `BookResult`（推荐）使其可被下游消费。

### C. CLI 缓存管理命令（建议：**建议**）

当前 `KZOCR_CLEAR_CACHE` 是环境变量，用户需要知道它的存在。建议添加 CLI 选项：
```
kzocr --clear-cache ocr ...
```

不阻塞实施但降低用户认知门槛。

## 5. 整体评估总结

### 优势

- **用户问题真、方案准**：从真实项目（TOC OCR 970 页实战）提炼，非空想
- **渐进式交付**：P0→P1→P2→P3 的阶梯确保每个阶段都有独立可交付的价值
- **评审吸收充分**：8 轮评审（架构/工程/安全/测试/领域/性能）的反馈已融入
- **测试覆盖面合理**：D1 的 32–42 用例覆盖了异常路径的完整组合
- **风险意识明确**：实施注意事项 7 项点出了适配器兼容性、CI L3 冲突、`_run_real` 不覆盖等风险

### 风险

| 风险 | 级别 | 缓解 |
|------|------|------|
| D2+D3 同一函数修改，分 PR 的 merge 冲突 | 中 | 建议同人实施（计划已标注）；D2 先提取子函数 |
| C1 L3 修改破坏现有 leakage 测试 | 低 | 计划已注明调整测试 |
| 适配器 `max_tokens` 兼容性未确认 | 中 | 需要在 D1/D2 编码前确认 PaddleOCR-VL 和 SenseNova 适配器状态 |
| `Retry-After` header 实际不可用 | 低 | 文档已有降级方案：纯指数退避 |

### 是否准备好实施

**基本准备好**。条件第 4 节的 A 和 B 应在 D2 编码阶段一并完成（不产生额外开发轮次），C 可接受延迟到 D3 阶段。

## 6. 裁决条件（3 项）

**条件 1（D2）：** 实现 `_process_vlm_page` 时，每次重试尝试前输出带有尝试次数和失败原因的日志消息。用户在 CLI 上能看到"第 X 页 API 失败，N 秒后重试（attempt m/3）"。

**条件 2（D2）：** 实现 `failed_pages` 跟踪后，在 `_run_vlm` 结束时输出汇总行（或扩展 `BookResult`），包含成功/失败页数和失败页面明细。

**条件 3（D3 阶段）：** 在 CLI 入口添加 `--clear-cache` 选项作为 `KZOCR_CLEAR_CACHE` 暴露的补充方式（一个文件的一行 argparse）。
