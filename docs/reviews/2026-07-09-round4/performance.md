# KZOCR 统一 OCR 引擎架构方案 —— 性能与资源评审（round4 · v0.2）

> 评审角色：性能与资源评审专家
> 评审对象：`docs/plans/ocr-engine-unification.md`（v0.2，已吸收 round3 修订）
> 重点：核查 round3 性能问题（P1–P8 / I8）在 v0.2 是否被**真闭合**
> 环境基准（沿用 round3 实测）：本机 **无 GPU**；12 核 CPU；可用内存约 6 GiB；llama-server 监听 `127.0.0.1:8080`，但 `config.vlm_port=18080`（端口不一致，见 P5）。
> 范围：仅调查与文档评审，未修改主方案 / 代码。

---

## 一、结论

**有条件通过（Conditional Pass）。**

v0.2 已把 round3 缺失的「性能预算」正式写入架构（§3 H0-E），`TOTAL_TIMEOUT` / 单页超时 / 并发上限 / 重试熔断 / 字形进程内镜像 / consensus 硬约束 全部具备，方向上 round3 的 P1/P3/P4/P5/P6 均已被**真闭合**。**但闭合度不是 100%**，存在两处必须在进入实现前修正的硬伤：

1. **预算阈值内部不自洽（最严重）**：`KZOCR_TOTAL_TIMEOUT=7200s`（并发 1）与 `KZOCR_MAX_PAGES=500` 在 VLM 单页时延下数学上互斥——7200/120s ≈ **60 页**即触总预算，500 页上限形同虚设，且任何 60 页以上的书在单本运行内**永远跑不完**。双闸退化为单闸，且把"大书须拆分"的边界压到不现实的 ~60 页/次。
2. **新引入 `AdapterPageResult.crop_img` 内存反模式**：若每页裁剪图以 `np.ndarray` 长期驻留于 `AdapterPageResult`，500 页 ≈ 2+ GB，恰恰击穿 v0.2 自己主张的"内存闸"，且 run.py 原实现只留文本、不持有图像（无此瓶颈）。

另有一处**自洽性反向**：§3 默认「无 GPU → 本地非视觉 PaddleOCR」，与 round3「无 GPU 下唯一可行路径是 VLM 整页推理」的结论相反，会把近乎不可用的逐行 PaddleOCR 设为默认，重演长尾慢路径。

上述三处均为**阈值/契约/默认值的修正**，不需重设计架构，故判「有条件通过」而非「需大修」。

## 二、round3 性能问题闭合度（逐条）

判定口径：✅ 已闭合 / 🟡 部分闭合 / ❌ 未闭合。

### P1 — consensus 多引擎并行在无 GPU 下不成立 → ✅ 已闭合
- v0.2 §3（L158-159）：`strategy.mode` 默认 `"single"`（硬约束，采纳假设 4）；`consensus` 仅 opt-in；**无 GPU 且全本地 CPU 引擎 consensus → 拒绝启动并告警**；仅含云端视觉引擎时允许 consensus 且 **N≤2**。
- 可执行性：由 `probe_environment()` + 纯函数 `select_adapters(probe, strategy, registry)` 在路由期强制裁决，CI 可注入 `ProbeResult` 确定性验证。与 §3 单引擎默认**自洽**（refusal 仅在显式 `mode="consensus"` 时触发）。
- 与 round3 B1 建议一字不差地对齐。

### P2 — VLM 无总超时 / 并发 / 资源预算 → 🟡 部分闭合
- 已补机制：单页 `timeout` VLM 120s / SenseNova 90s（对齐 B3）；`KZOCR_TOTAL_TIMEOUT=7200s` wall-clock 总预算，到点停后续页、已识别归档、未识别推 HumanGate（对齐 B4）；`KZOCR_MAX_PAGES=500`（对齐 B5）；`KZOCR_MAX_CONCURRENCY=1`（含云端≤2）（对齐 B2）。
- **但阈值不自洽**（见 §三-1）：总预算 7200s 在并发 1、单页 ~120s 下仅容 ~60 页，与 500 页上限矛盾，`MAX_PAGES` 实际不可达。机制对了，数字没对齐。
- 还缺一项实现约束：总预算必须在**每次（含重试）页尝试前后**复查，否则单页重试+退避会偷偷超出 7200s。v0.2 未明确"预算检查点位置"。

### P3 — 静默回退假数据（掩盖资源误用）→ ✅ 已闭合
- v0.2 §5/§9：降级/桩数据强制 `is_mock` 透传；`is_mock=True` 时归档/推送显 ERROR 且**阻断 publish**（对齐 I7 + round2 H8）；失败路径转 `HumanGate` 而非 `build_mock_book`。
- 资源误用语义已消除：不再"秒回假数据且掩盖上游不可用"。

### P4 — 无重试 / 退避 / 熔断，长尾页静默丢 → ✅ 已闭合
- v0.2 §3（L162）：`KZOCR_PAGE_RETRIES=2` + 退避；**同引擎连续 2 次超时熔断**本剩余页转 UNCERTAIN/HumanGate；明文"**禁止静默丢页**"。
- 直接对应 B6。run.py 的 `except → continue` 静默丢页被架构级禁令取代。
- 补强：熔断后剩余页免调用，天然防止长尾拖死整本（与总预算叠加双保险）。

### P5 — 端口 / 实例不一致导致重复起 llama-server → ✅ 已闭合
- v0.2 §3（L165）：统一 llama-server **端口真相源**，`auto_start` 前用**配置端口探测**（解决 `18080` vs `:8080`）；**可用内存 < 8GiB 时禁止 `auto_start`**，要求外部预起并告警。
- 直接消除"重复拉服务=双倍内存/CPU"的 round3 实测风险。

### P6 — 字形校验逐字查库隐藏瓶颈 → ✅ 已闭合（执行形态受约束）
- v0.2 §4.1（L173）：启动时一次性进程内镜像白名单+异体+繁简映射+**预计算** `confusion_set.json`+罕见中医候选字表；明文"**禁止逐字符查外部库（性能 P6）**"；`KZOCR_TERM_KB_PATH` 仅可选叠加且须受控目录校验。
- O(字符数) 校验，大书可忽略，瓶颈消除。
- 残留小风险：未规定"可选 `term_kb` 叠加"也必须先镜像进内存、禁止逐字符查——若日后有人让 term_kb 增强走在线查询，P6 会复活。建议补一句"任何 KB 增强一律先镜像，禁止在线查询"。

### P7 — 双页上下文对下一页完整重复渲染 → ❌ 未闭合（低严重）
- v0.2 §7 阶段 1 仅说把 `_pdf_page_to_numpy`/`_crop_to_body` 下沉到 `_common.py`，**未采纳 round3 B8 的"渲染一次缓存、上下文切片复用"**。
- 影响低（fitz 渲染毫秒级），但 `list(doc)` 全物化 + 每页重复渲染下一页顶 15% → 内存峰值翻倍仍在。建议阶段 1 顺手修。

### P8 — `list(doc)` 全量物化大书 → 🟡 部分闭合（低严重）
- v0.2 未显式采纳 B9「流式 + 仅缓存下一页」。但 `KZOCR_MAX_PAGES` 由 2000→**500** 把内存峰值上限砍掉 3/4，P8 被**上限削减缓解**而非根治。
- 500 页 Page 对象 + 双页重复渲染仍可能占数百 MB~1GB，在 6GiB 可用机上偏紧。建议阶段 1 落地流式，否则"内存闸"靠的是硬截断而非渐进治理。

### I8 — 性能预算整体缺失（无 GPU consensus + VLM 无总超时/并发/熔断）→ 🟡 部分闭合
I8 是 P1–P4 的汇总项。结论：预算**框架已落地**（从"零"到"有 KZOCR_* 全套预算变量 + 熔断 + 进程内 KB"），这是 round3 头号缺口的根本闭合；但**两个阈值缺陷（P2 不自洽、默认引擎反向）使框架在真实大书上会行为异常**，故判部分闭合而非全绿。

## 三、v0.2 新引入的性能问题

### N1（严重）— `KZOCR_TOTAL_TIMEOUT` 与 `KZOCR_MAX_PAGES` 阈值不自洽
- 证据：并发 1、单页 120s 上限（实际 VLM CPU 典型 30–60s、最坏 120s）。
  - 最坏：7200 / 120 = **60 页**触总预算。
  - 典型：7200 / 45 ≈ **160 页**触总预算。
  - 即无论哪种，500 页上限**永远先被总预算卡死**，`MAX_PAGES=500` 的"时间+内存双闸"退化成"只有时间闸"，且把大书拆分粒度压到不现实的 ~60–160 页/次（一本 500 页本草书需 3–8 次运行）。
- 根因：两个闸各自拍脑袋定数，未以"假定单页时延 × 页数"反推。
- 影响：① `MAX_PAGES=500` 名存实亡；② 真实大书**单本不可完成**，与"可归档"原则冲突；③ 运维对"能跑多少页"无正确预期。
- 修正见 §四-1。

### N2（中）— `AdapterPageResult.crop_img: np.ndarray` 长期驻留内存
- 证据：§2.1（L101）`crop_img: np.ndarray | None`，供 recheck/UX；§5（L202）HumanGate 行带"原图裁剪（crop_img 或路径+bbox）"。若 Router→GlyphVerifier→Archiver 全程持有每页 `np.ndarray`：
  - 150 DPI 裁剪页 ≈ 1000×1500×3 ≈ **4.5 MB/页**；500 页 ≈ **2.25 GB**，直接击穿 v0.2 自己主张的内存闸。
- 根因：契约把"瞬时 recheck 用图"与"持久 UX 用图"混为一谈，且默认类型是 `ndarray` 而非路径/bbox。
- 修正：契约层 `crop_img` 仅作**瞬时/可选**字段；持久化一律走"路径+bbox"或落盘的 `crop_path`，仅当 `VisionRecheck`/HumanGate 真正需要时按需从 PDF 重新渲染（渲染本就毫秒级，见 P7）。

### N3（中）— §3 默认引擎选择与 round3 性能结论反向
- 证据：§3（L158）"无 GPU → 本地非视觉 PaddleOCR"；而 §2.3 `PaddleOCRVl16Adapter` 标注"默认禁用，`auto_start` 须显绑 127.0.0.1"。
- 矛盾：round3 实测明确"PaddleOCR CPU 逐行 1–2s/行 **几乎不可用**，无 GPU 下唯一可行路径是 VLM 整页推理（PaddleOCR-VL-1.6 via llama-server）"。v0.2 却把**近乎不可用的 PaddleOCR** 设为无 GPU 默认、把**唯一可行 VLM** 默认禁用 → 开箱即走慢路径，重演长尾。
- 修正：默认探测顺序应为「有 llama-server/可 auto_start → VLM（PaddleOCR-VL-1.6）」优先于 PaddleOCR；或至少把 VLM 设为无 GPU 首选、PaddleOCR 降为无 VLM 时的兜底。

### N4（低）— 进程内 KB 启动耗时 × 每 CLI 重探
- 证据：§3（L164）"CLI 每次运行重新 probe"；§4.1 启动时一次性镜像白名单+异体+混淆集+可选 `term_kb`。
- 风险：KB 加载为进程级一次性成本，单本可接受；但若批量多书、每书一次 CLI 调用，KB 重复加载累加。且若 `KZOCR_TERM_KB_PATH` 指向大词库，启动时延上升。
- 修正：明确 KB 加载预算（目标 <1s 精简集）；大 `term_kb` 叠加须先限制规模或异步；批量场景建议常驻进程而非逐本 CLI。

### N5（低）— 客户端 120s 超时不能取消服务端生成
- 证据：客户端 HTTP `timeout=120` 会中断请求，但 llama-server 仍在后台跑完该页生成（孤儿任务）。配合"连续 2 次超时熔断"，若熔断前已发 2 个孤儿生成，会短暂空耗 CPU。
- 修正：熔断后向 llama-server 发取消/或依赖单实例串行自然消化；文档注明此边角，不必改架构。

## 四、改进建议（含阈值是否要调）

### 1. 重算并自洽化双闸（修正 N1，必做）
以"假定单页时延"反推其一，二选一落地到 §3：
- **方案 A（保时间闸、缩页数闸，推荐）**：保留 `TOTAL_TIMEOUT=7200s`；把 `KZOCR_MAX_PAGES` 改为与典型时延对齐的值——若 VLM 典型 45s/页，取 `7200/45 ≈ 160`，圆整为 **`KZOCR_MAX_PAGES=150`**；并明文"`TOTAL_TIMEOUT` 是真实 SLA 上限，`MAX_PAGES` 仅防极端快页下的内存尖峰"。
- **方案 B（保 500 页、放大时间闸）**：若业务确有 500 页单本需求，则 `TOTAL_TIMEOUT` 须 ≈ `500 × (典型时延+重试余量)`。按 45s×(1+20% 重试) ≈ 27h，取 **`KZOCR_TOTAL_TIMEOUT=86400s`(24h)**，并配套"24h 占用机器"的运维须知（与"防占机数天"目标需重新权衡）。
- **建议采用方案 A**：500 页中医书本就稀少，拆分到 ~150 页/次更符合"防占机"初衷；同时把"典型单页时延"作为显式假设写进 §3（如 `KZOCR_ASSUMED_PAGE_LATENCY=45s`），未来换 GPU/换引擎时据此重算双闸。
- 另：明确**预算检查点在每次页尝试（含重试）前后各查一次**，超时即中止后续页。

### 2. `crop_img` 契约改路径/bbox（修正 N2，必做）
- §2.1 `AdapterPageResult.crop_img` 增加注释："瞬时可选，不耐久驻留；持久 UX/recheck 一律经 `crop_path: str | None` + `bbox`"，把 §5 HumanGate 的"原图裁剪"明确为路径优先。视觉适配器默认不回填 `ndarray`，仅在 recheck 调用时临时渲染。

### 3. 反向默认引擎（修正 N3，必做）
- §3 默认链改为：无 GPU 时首选 VLM(PaddleOCR-VL-1.6)，PaddleOCR(本地非视觉) 仅作"无 VLM/无 llama-server"兜底；与 §2.3 的"VLM 默认禁用"同步放开 `auto_start` 端口绑定约束（仍须绑 127.0.0.1）。

### 4. 采纳 P7/P8 的流式+去重渲染（建议，低优先）
- 阶段 1 下沉 `_common.py` 时顺手：渲染一次缓存到数组、上下文切片复用（修 P7）；`list(doc)` 改流式 + 仅缓存"下一页"（修 P8），使内存闸由渐进治理而非硬截断实现。

### 5. 阈值微调建议（其余维持 round3 推荐值）
- 单页 VLM 120s / SenseNova 90s：**维持**（合理，留重试余量）。
- `KZOCR_PAGE_RETRIES=2` + 退避 2s/4s：**维持**（对齐 B6）。
- "连续 2 次超时熔断"：**维持**，但建议补"熔断后本剩余页转 UNCERTAIN 并记审计，而非直接丢弃"，与"禁止静默丢页"完全闭环。
- `KZOCR_MAX_CONCURRENCY=1`(含云端≤2)：**维持**；建议 §3 补一句"纯本地 VLM 时并发恒为 1，吞吐量受单 llama-server 实例上限约束，故大书须靠双闸拆分而非并发提速"。

## 五、对假设 4 裁决的再确认

round3 汇总 §4 裁决：**采纳（默认 single，consensus 仅 opt-in，提升为硬约束）：无 GPU 全本地 consensus 拒绝启动；含云端时 N≤2；单引擎以 UNKNOWN/低置信补触发**。

**再确认结论：完全支持并确认 v0.2 已正确落地该裁决，且较 round3 更严格、更可执行。**
- v0.2 §3 把 `strategy.mode` 默认写死 `"single"` 并标"硬约束"，比 round3 原"默认 single"语气更强、可 CI 校验；
- "无 GPU 全本地 consensus 拒绝启动并告警" 由 `select_adapters` 纯函数强制，可执行、可测；
- "含云端视觉引擎时 N≤2" 与 `KZOCR_MAX_CONCURRENCY≤2` 自洽；
- "单引擎以 UNKNOWN/低置信补触发" 在 §3（L160）与 §4.2/§5 的 `UNKNOWN→HumanGate` 闭环对齐（同时闭合了 I6 漏放）。
- 唯一连带提醒：假设 4 的"无 GPU 默认 single"依赖于**默认引擎选对**——见 N3，若默认回落到 PaddleOCR 逐行，则"single 也慢"会削弱假设 4 的性能收益。修正 N3 后假设 4 方才完整兑现。

**一句话**：假设 4 裁决本身无懈可击，v0.2 落地正确；但需 N1（双闸阈值自洽）、N2（crop_img 内存）、N3（默认引擎反向）三处配套修正，否则性能预算框架在真实大书上会行为失准。
