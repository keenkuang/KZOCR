# KZOCR 统一 OCR 引擎架构方案 —— 性能与资源评审（round3）

> 评审角色：性能与资源评审专家
> 评审对象：`docs/plans/ocr-engine-unification.md` v0.1（重点第 1/3/4 章）
> 环境基准（实测）：本机 **无 GPU**；12 核 CPU；内存 15 GiB，可用约 6 GiB；llama-server 当前监听 `127.0.0.1:8080`，但 `config.vlm_port=18080` 且适配器默认端口同为 `18080`（端口不一致，见 P5）。
> 既有约束：真实 kimi 逐行 OCR（PaddleOCR CPU 1–2 s/行）几乎不可用；无 GPU 下最佳出路是 VLM 整页推理（PaddleOCR-VL-1.6 via llama-server），但需先起本地服务；云端视觉需 key 且涉数据出境。

---

## 一、结论

**方案在"可切换/可归档/质量门"的方向上正确，但性能与资源设计严重缺位，尤其是无 GPU 这一既定前提未被贯彻到架构决策里。**

四个核心结论：

1. **`consensus` 模式在无 GPU 下不应作为可默认启用的能力**。两个 CPU 密集型引擎（PaddleOCR-VL 占 10 线程 + PaddleOCR 逐行）并行，只会在 12 核上互相抢占，端到端反而更慢且更易 OOM。**方案假设 4 的"默认 single、consensus 可选"结论正确，应提升为强制约束而非默认。**
2. **VLM 整页推理是无 GPU 下唯一可行路径，但当前实现是纯串行、单请求、无并发、无总超时、无整体资源预算**。单 llama-server 实例（`--parallel` 默认为 1）下，客户端并发请求只会被队列化，无任何加速。
3. **`KZOCR_MAX_PAGES=2000` 不是 DoS 防护，是内存防护**。2000 页在 CPU 上以 30–120 s/页估算 = 16–66 小时，远超任何合理 SLA；它既不防时间耗尽，也不防 llama-server 长占资源。
4. **字形校验成本可控，但前提是"内存化、批量化、启动期建索引"**；若按字面"逐字比对知识库"做逐字符外部查询（term_kb/RuntimeDB），大书会成为隐藏瓶颈。

**总体立场：方案在第 8 章假设上多数成立，但需把"性能预算/超时/并发上限"作为第 0 优先级补进架构，否则 single 模式也会因长尾页/无总超时而失控。**

## 二、关键问题（按严重度）

### [严重] P1 — `consensus` 多引擎并行在无 GPU 下不成立
- **位置**：方案 §3 策略 B（多引擎共识）；`run.py` 现有三路分支无并发设计。
- **事实**：PaddleOCR-VL-1.6 在 CPU 上由 llama-server `-t 10` 占满 ~10 核（`paddleocr_vl16_adapter.py:94`），模型单实例、`--parallel` 默认 1。PaddleOCR CPU 逐行 1–2 s/行。两引擎并行时：VLM 被挤出核 → 单页从 ~30 s 退化到数分钟；PaddleOCR 仅剩 2 核 → 更慢。共识比对还要缓存 N 份 `engine_texts`（§3/§4.3），内存与序列化成本叠加。
- **影响**：端到端时延翻倍、OOM 风险上升、结果未必更准（VLM 已是最优本地源）。
- **建议**：见改进建议 B1/B2。

### [严重] P2 — VLM 推理无总超时、无并发、无资源预算，单页可拖垮整本
- **位置**：`paddleocr_vl16_adapter.py:192`（`timeout=600`）、`run.py:451` 主循环、`KZOCR_MAX_PAGES`。
- **事实**：
  - 每页一次串行 HTTP POST，超时 600 s；若某页卡死，最坏 600 s/页。
  - 纯顺序，无并发、无批处理；llama-server 单实例无法并行消化并发请求。
  - 全链路**没有 wall-clock 总预算**。即使 `MAX_PAGES=2000` 生效，也只是 `list(doc)` 截断（memory guard），不约束总时长。
- **影响**：用户无法预期完成时间；单本大书可占机数小时至数天，期间 llama-server 独占资源，其他任务饿死。
- **建议**：见 B3/B4/B5。

### [严重] P3 — 重降级路径"静默回退假数据"放大资源误用（兼性能语义问题）
- **位置**：`run.py:39-51`（VLM/真实失败 → `build_mock_book` 返回假数据，仅 print 警告）。
- **事实**：VLM 全部失败后不报错而返回占位假数据（round2 的 H8 已列）。从性能视角看，它让一次本应失败/转人工的长任务"秒回"，但产出是假的，且**掩盖了上游引擎不可用的事实**，使资源投入（起 llama-server、渲染整本）变成纯浪费。
- **影响**：运维侧误判"任务成功"，重复投入算力。
- **建议**：降级必须显式 `is_mock` 标记 + 非零退出/异常上抛（已被 round2 H8 要求，此处重申性能影响）；失败应转 `HumanGate` 而非假数据。

### [中] P4 — 渲染/识别"无重试、无退避、无熔断"，长尾页直接丢
- **位置**：`run.py:470-472`（`except → continue` 跳过整页）、`paddleocr_vl16_adapter.py:198-200`（异常返回 `""`）。
- **事实**：单页失败即 `continue` 丢弃，无重试、无退避、无熔断。VLM 偶发超时（网络/上下文）会被直接当成"空页"，损失信息且无补救。
- **影响**：大书长尾丢页率高；"降级"未做智能处理，只是丢弃。
- **建议**：见 B6（每页重试预算 + 熔断 + 丢页审计）。

### [中] P5 — 端口/实例配置不一致，可能重复起 llama-server（资源浪费）
- **位置**：`config.vlm_port=18080`（`kzocr/config.py`）、`paddleocr_vl16_adapter.py:29`（`_DEFAULT_PORT=18080`）、实际监听 `:8080`。
- **事实**：适配器按 18080 探测，`_is_server_running()` 返回 False → `auto_start=True` 会再拉一个 llama-server（模型加载 60 s+，占用数 GiB 内存，且 `-t 10` 抢核）。本就紧张的 6 GiB 可用内存可能 OOM-swap。
- **影响**：重复起服务 = 双倍内存/CPU，启动慢，且两个实例端口不同互相无感知。
- **建议**：统一端口配置（单一真相源）；`auto_start` 前必须用"配置端口"探测；或要求外部预起服务、关闭 `auto_start`。

### [中] P6 — 字形校验"逐字比对知识库"若按字面实现，大书有隐藏瓶颈
- **位置**：方案 §4.2（逐字校验 1–4 步）、§4.1（基于 term_kb/RuntimeDB）。
- **事实**：方案说"逐字/逐行比对字形知识库"。若每字符都走：
  - (a) 外部查询（Postgres/RuntimeDB `term_kb`）→ 大书百万级字符 = 百万次查询，灾难；
  - (b) 内存 `set` 成员判定 → 每字符 O(1)，整本仅 O(字符数)，可接受（CJK 全集 + 中医词典）。
  - 另含"形似混淆（未/末、已/己）"相似图判定，若在线计算编辑距离/相似度则偏重；若预计算相似表则廉价。
- **影响**：实现方式决定是"忽略不计"还是"大书主瓶颈"。方案未约束实现形态。
- **建议**：见 B7（强制内存化 + 批量化 + 预计算相似表，禁止逐字符外部查询）。

### [低] P7 — 两页上下文对下一页做了**完整重复渲染**
- **位置**：`run.py:459-464`（SenseNova 双页上下文）。
- **事实**：为取"下页顶部 15%"，代码 `_pdf_page_to_numpy(all_pages[i+1])` 全分辨率 150 DPI 渲染后再切片。下一轮主循环又会再次渲染同一页 → 每页（除末页）被渲染 2 次。
- **影响**：渲染本身在 fitz 下仅毫秒级（非瓶颈），但属无谓浪费，且 `next_full` 当轮释放前又占一份内存；在 `list(doc)` 全物化前提下内存峰值更高。
- **建议**：渲染一次缓存到数组/字典，上下文切片复用。

### [低] P8 — `list(doc)` 全量物化大书
- **位置**：`run.py:443`（`all_pages = list(doc)`）。
- **事实**：round2 已标记。2000 页 Page 对象全驻内存，配合双页重复渲染（P7）内存峰值翻倍。
- **影响**：大书内存尖峰；与 `MAX_PAGES` 一起才勉强可控。
- **建议**：流式 `for page in doc` + 按需渲染 + 仅缓存"下一页"用于上下文（见 P7）。

## 三、改进建议（含具体阈值）

### B1 — consensus 默认关闭，GPU 缺失时设为硬约束
- `strategy.mode` 默认 **`"single"`**（方案假设 4 立场正确，建议写为强制而非默认）。
- 当 `probe_environment()` 判定 **无 GPU** 且 `mode == "consensus"` 时：
  - 若并发引擎全部为本地 CPU 型（local-nonvision / local-vision）→ **拒绝启动并告警**，提示"无 GPU 下 consensus 不经济，请改用 single 或纳入一个云端引擎"。
  - 仅在"至少包含一个云端视觉引擎（offload CPU）"时才允许 consensus，且 `N` 硬上限 **2**。
- 理由：CPU 上两个本地引擎并行是纯粹的内耗。

### B2 — 并发上限与资源争用控制
- 引入 `KZOCR_MAX_CONCURRENCY`（默认 `1`；有 GPU 或含云端引擎时可 ≥2）。
- 客户端并发必须与 llama-server `--parallel` 对齐：无 GPU 下建议 `--parallel 2~3`（`-t` 相应降到 6~8，留核给渲染/系统），并发请求数 ≤ `--parallel`，否则徒增队列与内存。
- 单本同时存活的"本地视觉引擎实例"上限 = **1**（VLM 模型单实例最经济）。

### B3 — 单页超时收紧
- `PaddleOCRVl16Adapter` 每页 `timeout` 由 **600 s 降到 120 s**；SenseNova 由 180 s 降到 90 s。
- 超过阈值即判该页失败（转 UNCERTAIN / 重试预算内重试），不无限等待。

### B4 — 增加 wall-clock 总预算（SLA 兜底）
- 新增 `KZOCR_TOTAL_TIMEOUT`（默认 **7200 s = 2 h** / 单本）。到达后：停止后续页，已识别页正常归档，未识别页统一推送 `HumanGate`（标注 `timeout`）。
- 新增 `KZOCR_ETA_LOG_EVERY`（默认每 5 页，沿用 `run.py:452` 的日志节奏）输出进度与预计剩余时间，使用户有 SLA 预期。

### B5 — 重设页数上限语义
- `KZOCR_MAX_PAGES` 默认由 **2000 降到 500**；它应当与 `KZOCR_TOTAL_TIMEOUT` 配套作为"时间+内存"双闸，而非仅内存闸。文档明确：该值是"防失控上限"，正常书籍不应触顶，触顶即告警并建议拆分任务。

### B6 — 每页重试 + 熔断 + 丢页审计
- 每页失败重试预算 `KZOCR_PAGE_RETRIES=2`，退避 `2^s`（2/4 s）。
- 同一引擎连续 2 次超时 → **熔断**该引擎本剩余页（不再调用，转 UNCERTAIN/HumanGate），避免长尾拖死整本。
- 任何被跳过/失败的页必须记审计（页码、引擎、原因），最终汇总到 HumanGate，禁止静默丢弃（呼应 P4、H8）。

### B7 — 字形校验实现形态约束
- `GlyphVerifier` 启动时**一次性**把字形白名单 + 中医词典 + 异体/繁简映射 + 预计算的"形似混淆表"载入进程内 `set` / `dict`，**禁止逐字符外部查询**。
- 校验以"集合成员判定 + 预计算相似表命中"为主，O(字符数)，大书可忽略。
- VLM 无字级置信度 → 退化为行级/整页级（方案 §4.1 已述），靠共识补强；不引入再识别视觉模型（认同假设 2，避免二次 VLM 成本）。

### B8 — 渲染与 DPI 预算
- 150 DPI 渲染与 `_crop_to_body`（numpy 投影，O(h·w)）本身**非瓶颈**（毫秒级），但建议把 DPI 外置为 `KZOCR_DPI`（默认 150；无 GPU 下可降到 **120** 节约约 36% 像素、减轻 VLM 编码器负载，需标注准确率折损）。
- 修复 P7 双页重复渲染：渲染一次缓存，上下文切片复用；并把 `list(doc)` 改为流式（B9）。

### B9 — 内存与实例治理
- 流式处理 + 仅缓存"下一页"（用于上下文）；避免 `list(doc)` 全物化（P8）。
- 统一 llama-server 端口（单一真相源），`auto_start` 前用配置端口探测（P5）；内存紧张（可用 < 8 GiB）时禁止 `auto_start`，要求外部预起并告警。
- 监控 llama-server RSS：模型 + `-c 131072` 上下文在 6 GiB 可用机上偏紧，建议 `-c` 按需下调（整页 OCR 实际用不到 128K 上下文）。

## 四、对第 8 章假设立场

| 假设 | 本评审立场 | 理由 |
|---|---|---|
| **假设 4（consensus 成本：是否默认仅 single）** | **强烈支持默认 single，并提升为"无 GPU 下 consensus 仅允许含云端引擎且 N≤2"的硬约束** | P1：两个本地 CPU 引擎并行纯内耗。方案原表述"默认 single、consensus 可选"过于软弱，应明确禁止无 GPU 的全本地 consensus。 |
| **假设 2（暂不加独立再识别视觉模型）** | **支持** | P6/B7：再识别等于对 FAIL/UNCERTAIN 行二次跑 VLM，CPU 下成本爆炸；字典+置信度+共识已够，且 VLM 无字级置信度本就靠共识补。 |
| **假设 6（字形知识库复用 kimi `term_kb`/RuntimeDB）** | **同意复用来源，但必须进程内镜像，禁止在线查询** | P6：若直接逐字符查 RuntimeDB，大书是主瓶颈。建议 KZOCR 内置精简白名单 + 启动时从 `term_kb` 同步到内存。 |
| **假设 1（字形校验机制为主）** | **同意，但补"性能预算"作为前提** | 校验本身廉价，真正风险在 §3/§4 缺总超时/并发/熔断；建议把性能预算写入架构第 0 优先级。 |
| **假设 3（方剂库归属）/ 假设 5（配置存放）** | 与性能无关，不持立场 | — |

**一句话立场**：方案第 8 章对"无 GPU 下应 single"的判断是对的，但低估了 VLM 这条唯一可行路径自身的资源失控风险——**必须补"单页 120 s 超时 + 单本 2 h 总预算 + 并发≤1（本地）/≤2（含云端）+ 熔断 + 内存化字形校验"**，否则即便 single 模式也会因长尾页与无总超时在大书上失控。
