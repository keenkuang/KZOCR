## 一、架构师 (Architect) 视角

### ✅ 正面

1. **后处理分离模式合理**：`enrich_book_result()` 作为 E4 后置步骤而非嵌入 E4 内部，与 v0.7 概览 §7「委派模式，旧签名保留」的迁移策略一致。不会污染 `orchestrate_book` 的主循环逻辑。

2. **模块定位正确**：`kzocr/engine/toc.py` 依赖 `types.py`（通用层），不依赖 `scheduler/` 或 `engines/`，符合「`engine/` 依赖 `engines/`，反之不成立」的现有约定。

3. **类型设计自洽**：`TocTree.entries` 为顶级条目数组、`TocEntry.sub_entries` 递归嵌套，天然支持非均匀深度（有些章 5 层、有些 2 层），符合中医古籍实际。

### ⚠️ 风险

1. **`enrich_book_result` 返回值语义模糊**（§4 `enrich_book_result`）：

   ```python
   def enrich_book_result(result: BookResult) -> BookResult:
       result.toc = build_toc(texts)
       return result
   ```

   该函数**就地修改**传入的 `BookResult.toc` 后返回同一对象引用——调用方可能误以为是深拷贝出的新对象。如后续有并行或缓存场景，副作用易引发隐晦 bug。

2. **`TocTree.book` 与 `BookResult.title` 重复**（§2 `TocTree`）：

   ```python
   @dataclass
   class TocTree:
       book: str = ""
       total_pages: int = 0
   ```

   `TocTree.book` 的语义与 `BookResult.title` 重叠。`total_pages` 含义模糊——是书的 PDF 总页数，还是 TOC 部分占的页数？后者与 `discover_toc_pages` 的返回值直接冲突（该函数返回的是 TOC 页号列表）。

3. **未利用已有 `ParagraphResult.heading_level` 字段**（`types.py` 行 75）：

   `PageResult.paragraphs[].heading_level: Optional[int]` 已在数据模型中存在。E4 产出 `PageResult` 时会标注标题层级。TOC 抽取后应当与正文中的 `heading_level` 做交叉验证（例如：TOC 说 §1 在 p.1，但实际上 p.1 的 `heading_level=3` 而非 `heading_level=2` → 可能 TOC 层级推导有误）。当前计划完全未提及此交叉验证。

### 💡 建议改进

- **方案 A（推荐）**：`enrich_book_result` 改为显式返回新对象或纯函数风格。至少加 docstring 注明「就地修改并返回同一引用」。
- **方案 B**：去掉 `TocTree.book` 字段（由 `BookResult.title` 承载）；`total_pages` 改名为 `cover_page_count` 或干脆去掉（调用方可从 `discover_toc_pages` 返回值直接获知 TOC 页数）。
- 建议在 `build_toc` 后增加一个可选的交叉验证步骤（默认关闭），遍历 `BookResult.pages` 查找 `heading_level` 与 TOC 条目 page 是否对齐。

## 二、安全专家 (Security) 视角

### ✅ 正面

1. **无网络调用**：采用纯文本正则方案（不做 VLM），不依赖外部 API，无 API key 泄漏风险。
2. **输入有界**：`pages_text: list[str]` 来自 `BookResult.pages`，页数上限由 `max_pages` 约束（`orchestrator.py` 行 174：`getattr(config, "max_pages", 50)`），OOM 风险可控。

### ⚠️ 风险

1. **TOC 标题→文件路径的传播链未审计**：

   姊妹项目 `section_merger.py` 行 506 使用 `re.sub(r'[\\/:*?"<>|（）()\\s]', '', r["title"])[:30]` 将章节标题直接写入文件名。虽然清理了特殊字符，但 KZOCR 的 `TocEntry.title` 来自 OCR 文本，若 OCR 产出 `../../../etc` 或 `..\\..\\..\\boot` 等续字（C1 跨页泄漏），在未清理的场景下可能路径穿越。当前计划未说明 TOC 标题是否会被传递到文件系统操作中。

2. **正则 ReDoS 风险**：

   计划 §3 `parse_toc` 中正则匹配点分节号、分隔符（……/全角空格）、编号前缀(`(1)`/`①`)。如果一条 OCR 文本行很长（例如跨页泄漏导致上千字符拼接），复杂的正则组合可能导致回溯爆炸。姊妹项目 `section_merger.py` 行 34 的 `RECIPE_NO_PATTERN` 已使用 `{1,30}` 限制标题长度，但 `toc.py` 当前未设定量词上界。

### 💡 建议改进

- 在 `parse_toc` 的正则中添加显式长度上限（如 `{1,100}`），防止超长行引发 ReDoS。
- 在 `build_toc_tree` 和所有输出接口的 docstring 中注明：**输出仅用于结构化索引和元信息，不适用于直接拼接文件路径**。若后续需要将 TOC 标题用于文件名，须在调用处做路径合法性校验（参考 `engines/atomic.py` 的路径穿越防御模式）。

## 三、领域专家 (Domain) 视角

### ✅ 正面

1. **5 层层级结构贴近实际**：中医古籍目录确实常见 卷→科→章→节→方 的多级嵌套。`level=1` 对应"卷/门/科/部"、`level=5` 对应子条目如 `(1)` 的建模合理。

2. **SEO-BUG-001 经验采纳正确**：traedocu 章/节号误判致 33% 的问题已通过三级计数器自校验机制回应，这是领域经验的有效迁移。

### ⚠️ 风险

1. **目录页探测的 OCR 容错严重不足**（§3 `discover_toc_pages`）：

   ```python
   # 计划中的探测规则
   - 某页包含"目　录"、"目录"、"CONTENTS"等标题
   ```

   实际 OCR 产出中，"目"可能被识别为"日"、"自"（如"目录"→"日录"、"自录"），"录"可能为"彔"、"隶"。姊妹项目 `toc_parser.py` 使用 VLM（视觉理解）规避了这个问题，但 KZOCR 是文本方案，必须增加**模糊匹配**。此外：
   - 繁体"**目錄**"完全没有提及
   - 中医古籍目录标题的变体："**總目**"、"**卷首**"、"**綱目**"、"**條目**"、"**敘目**"
   - 部分古籍无显式目录标题，仅"**引用書目**"或"**凡例**"中隐含目录

2. **页码格式过于简化**（§3 `parse_toc`）：

   计划假设目录中的页码始终是阿拉伯数字。但中医古籍目录页码格式多样：
   - **中文数字**：一、二、三… 或 壹、贰、叁…
   - **叶码（面码）**：古籍常标记"叶"而非页，一叶 = 两页面
   - **年号纪年替代页码**：某些序跋目录无数字页码
   - **部分条目无页码**：卷首/卷末的附录条目经常无页码，应对策略未说明

3. **层级推导的"缩进"假设在 OCR 文本中不可靠**（§3 层级推导）：

   计划包含「缩进前缀判定层级」，但 **OCR 文本丢失空格/缩进信息**。OCR 引擎输出通常不会保留排版缩进（除非特别指定保留空格）。从 `PageResult.text`（聚合文本）中推断缩进几乎不可能。计划中的「缩进量 > 上一条 → level 递增」逻辑在文本模式下可能完全失效。

4. **level=3（节）的 §N 匹配过于狭窄**：

   中医古籍中节标题常表现为：
   - 「**第一節**」或「**第一节**」
   - 「**一、**」无 § 前缀
   - 「**（一）**」使用括号编号
   - 纯关键词标题如「**治感冒秘方**」无编号
   计划只列了 `§N` / `N.N` 两种模式，覆盖率偏低。

### 💡 建议改进

- **增加模糊目录页匹配模式**：使用编辑距离（Levenshtein）或 Unicode NFC 归一化匹配"目录"类标题。支持可配置的多标题关键词列表（"總目"、"綱目"、"目錄"、"CONTENTS"、"TABLE OF CONTENTS"）。
- **增加中文数字→阿拉伯数字转换**：集成 `zhon` 或自建字典处理一二三/壹贰叁页码。
- **缩进退化为"仅当有连续空格时使用"**：添加明确说明——文本式 TOC 解析中缩进探测可能不可用，主要依赖编号深度和关键词匹配。
- **扩展 §N 匹配为更通用的"节标题检测"**：增加 `^(第?[一二三四五六七八九十]+节|第?[0-9]+节|（[一二三四五六七八九十]+）) ` 等模式。

## 四、测试专家 (Test) 视角

### ✅ 正面

1. **层级覆盖较好**：3 层（卷→章→节）和 5 层（卷→章→节→小节→子条）都有测试用例。
2. **跨页合并有覆盖**：目录跨两页时合并逻辑的测试已列明。
3. **格式变体覆盖**：无缩进分隔符填充、双空格分隔符、繁简混用都有提及。

### ⚠️ 风险

1. **缺少 OCR 噪声场景测试**：

   这是 **最严重的测试缺口**。文本式 TOC 抽取的最大脆弱性来自 OCR 识别错误，但测试计划中所有 mock 用例都假设 OCR 文本完全正确。必须增加：
   - 目→日/自 混淆（"日录"→仍应触发发现）
   - 页码被识别为乱码（`1`→`I`、`8`→`&`）
   - 标题中重要关键字的 OCR 错误（"内科"→"闪科"、"外科"→"处科"）
   - 条目连续两行被合并为一行（缺换行）
   - 全角/半角数字混用

2. **缺少负向测试（False Positive）**：

   - 一本书**根本没有目录页**（常见于某些抄本、残本）→ 应返回 `None` 而非空树
   - 正文中出现 "目录" 二字但不是目录（例如书中标题为"**目录篇**"）→ 不应误触发
   - 附录中页码格式恰好匹配模式但并非目录条目
   - 仅有 `目　录` 标题行但无条目（计划 `line82` 说跳过无页码行，但缺少显式测试）

3. **缺少大目录压力测试**：

   `build_toc_tree` 用的是面包屑路径算法（每层一个当前节点），目录条目数过多时（例如某些全书目录有 500+ 条目、最深 5 层）需要确保：
   - 算法复杂度为 O(n) 而非 O(n²)
   - 递归深度可控（Python 默认递归限 1000，5 层安全，但间接递归需确认）
   - 大量同 level 条目（例如 200 个 level=5 条目连续出现）不会卡死

4. **`seen_recipe_nos` 去重逻辑测试缺失**：

   计划 §3 `build_toc_tree` 提到「遇到 section_no 重复 → 标记（勘误处理，可选）」，但测试计划中**没有对应用例**。姊妹项目经验表明编号重复是真实痛点（`section_merger.py` 行 23-29 的 `BOOK_ERRATA`），KZOCR 应当覆盖此分支。

### 💡 建议改进

- **新增测试组 "OCR 噪声 resilience"**：至少 3 个用例：a) OCR 将"目录"识别为"日录"仍触发发现；b) OCR 将页码数字混淆仍正确提取；c) 条目缺行合并后仍恢复。
- **新增测试组 "负向过滤"**：至少 2 个用例：a) 无目录页时返回 None；b) 正文含"目录"关键词但无目录结构时不误报。
- **新增测试组 "大目录压力"**：构建 500+ 条目 5 层深度的 mock 输入，验证 `build_toc_tree` 在 1 秒内完成。
- **新增测试组 "编号重复处理"**：构建含两个 `section_no="1"` 的 flat entries，验证 `build_toc_tree` 触发勘误标记（不崩溃）。
