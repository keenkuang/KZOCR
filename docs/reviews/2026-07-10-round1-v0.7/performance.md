# v0.7 自适应 OCR 引擎编排层 —— 性能评审

> 评审角色：性能工程师
> 评审对象：`docs/plans/ocr-engine-unification.v0.7.md`
> 参考实现：`kzocr/engine/run.py`（B6 MAX_PAGES / TOTAL_TIMEOUT 双闸、VLM 集成）
> 环境假设（沿袭已确认的基准）：本机**无 GPU**；12 核 CPU；可用内存约 6 GiB；引擎均经 BookPipeline 或本地 VLM 调用
> 范围：仅文档评审，未修改方案/代码

---

## 一、总体判断

**需大修后重审（Major Revision Required）。**

v0.7 在架构层面（注册中心 + 调度器 + 字形验证器 + 编排循环）方向正确，但存在 **3 处性能硬伤** 和 **2 处设计-实现矛盾**，直接导致：
1. 无 GPU 下 Tier 1 并行调度**不仅无收益，反而有害**（CPU 争抢 + 吞吐下降）；
2. 编排主循环缺失 B6 双闸检查，大书可耗尽资源；
3. 三级兜底最坏延迟 ≈ 190s/页 × 200 页 ≈ **10.5 小时**，且无配套预算保护；
4. benchmark 写盘策略未定义，存在 O(n²) 退化和 I/O 拥堵风险；
5. 内存基线取决于 `render_pages` 实现，风险不确定但可规避。

**修正方向集中于实现细节与预算契约，不涉及架构推倒重来。**

## 二、逐项分析

### 2.1 多引擎并行调用（Tier 内 2 引擎并行）—— 在无 GPU 下反收益

**证据：**

| Tier | 引擎类型 | 计算特性 | 并行效果 |
|------|---------|---------|---------|
| 1 | OCR 引擎（paddleocr/rapidocr/mineru/unirec） | 纯 CPU 密集 | 竞争相同 CPU 核 → wall-clock 不降反升 |
| 2 | 云端视觉 LLM（sensenova） | 网络 I/O 密集 | 并行有效，但计划限并发 1 |
| 3 | 本地 LLM（shizhengpt/paddleocr_vl16） | GPU 或 CPU 密集 | 无 GPU 时 paddleocr_vl16 不可用；shizhengpt CPU 推理极慢，并行无益 |

具体分析：

**Tier 1 在无 GPU 下的反收益：**
- 4 个 OCR 引擎均经 BookPipeline 调用，BookPipeline 自身含内部并行（LLM 校对等）。
- 启动 2 个 BookPipeline 实例意味着：双倍 CPython GIL 争抢 + 双倍进程内存（BookPipeline 加载模型词典约 200–500 MB/实例）。
- 假设单引擎单页 5s（CPU 密集），并行 2 个 → 每个分配到 ~50% CPU → 单个完成时间 ≈ 10s → max(10s, 10s) = **10s 并列比 5s 串行更慢**。
- 仅在引擎为 I/O 密集（如 picklereader 纯读盘）时并行有意义，但 Tier 1 引擎不符此特征。

**Tier 2 并行合理但未用：** 计划 max_concurrency=1，未利用并行优势。
**Tier 3 并行不可取：** 本地 LLM 的 CPU 推理已很慢，并发只会更慢。

**设计-实现矛盾：** 计划「现状 vs 目标」表声称「多引擎并行（同一页）」——但 E4 伪代码显示的是串行 `for engine in engines` 循环，并无并行执行语义。并行声言与串行伪代码矛盾。

**→ 结论：无 GPU 下 Tier 1 并行应默认关闭（并发=1），仅在 opt-in + 有 GPU 时允许并发 >1。**

### 2.2 调度器候选排序计算开销 —— 可忽略，但资源过滤可能成为真瓶颈

**排序计算分析：**

```
score = glyph_pass_rate × (1 / avg_latency)
```

- 每引擎一次浮点乘法除法：~5ns/引擎
- 9 引擎全排序：~45ns + Python 函数调用开销 ≈ 1–5μs
- 调用频次：每页每级（200 页 × 3 级 = 600 次/书）
- 总开销：600 × 5μs = **3ms/书 —— 可忽略**

**无需缓存**。9 个引擎的排序微秒级完成，引入缓存反而增加复杂度（缓存失效逻辑、一致性维护）。

**真正隐患：资源过滤中的外部调用**

`select_candidates` 第 3 步「资源过滤」包含：
- VRAM 足够 → 若涉及 CUDA API 调用（如 `torch.cuda.mem_get_info()`），耗时 1–5ms
- 网络可达 → 若使用 TCP connect 探测，耗时 50–200ms

若每次选择都执行这些检查，600 次 × 200ms = **120s/书 —— 不可接受**。

**→ 建议：** 资源检查仅在发生状态变更时执行（EngineProbe 探测周期/错误触发），候选选择时只读缓存状态位。

### 2.3 B6 双闸（MAX_PAGES + TOTAL_TIMEOUT）—— 编排层未继承，存在回归风险

**现状 `run.py` 的保护：**

| 保护 | 环境变量 | 默认值 | 检查时机 | 位置 |
|------|---------|--------|---------|------|
| 页数上限 | `KZOCR_MAX_PAGES` | 50 | 页面循环前（一次截断） | run.py:578 |
| 总时间预算 | `KZOCR_TOTAL_TIMEOUT` | 7200s | **每页**循环中检查 | run.py:592 |

**v0.7 编排主循环（E4 pseudo-code）的缺失：**

```
for page in render_pages(pdf_path, config):    ← 无 MAX_PAGES 截断
    ...
    engines = scheduler.select_candidates(..., budget)  ← Budget 对象存在但未在循环级别使用
    ...
```

- `render_pages` 可能渲染全部页（数百页）到内存
- `Budget(config)` 传入调度器做单次选择，但**循环级别无时间闸**——如果 Tier 1 两引擎都失败 → Tier 2 → Tier 3，这一页可能花掉 190s，但循环仍继续处理后续页
- 对比当前实现：`_run_vlm` 每页都检查 `elapsed > total_timeout`，超时立即 break

**风险：** 同一梯队的多引擎串行执行（伪代码中的 `for engine in engines`）会累加时间，而时间闸只在调度器入口检查「本次调用预算」而非「整本书已用预算」。**双闸实现降级为无保护。**

**→ 修复方向：** 编排主循环必须在 `for page` 入口处恢复双闸检查：
```python
for page in render_pages(...):
    budget.check_page_limit(page_num)     # ← 页数闸
    budget.check_time_budget(elapsed)     # ← 时间闸（每页检查）
    if budget.exhausted:
        break
```

### 2.4 三级兜底最坏情况延迟

**单页最坏路径（伪代码所示串行执行）：**

```
Tier 1:
  engine[0] run (OCR CPU, ~30s worst)
  → glyph verify (~0.01s)
  engine[1] run (OCR CPU, ~30s worst)
  → glyph verify (~0.01s)
Tier 2:
  engine[0] run (cloud VLM, ~10s worst)
  → glyph verify (~0.01s)
Tier 3:
  engine[0] run (local LLM CPU, ~120s worst)
  → glyph verify (~0.01s)
──────────────────────────────────────────
总计: ~190s / 页
```

**百页汇总：**

| 场景 | 单页延迟 | 100 页 | 200 页 |
|------|---------|--------|--------|
| Tier 1 成功（最优） | 5s | 500s (~8min) | 1000s (~17min) |
| 三级全走（最坏） | 190s | 19,000s (~5.3h) | 38,000s (~10.5h) |
| 当前 VLM 单路径（对比基线） | 30s | 3000s (~50min) | 6000s (~1.7h) |

**关键发现：** v0.7 的最坏情况比当前 VLM-only 路径慢 **6.3 倍**。在 `KZOCR_TOTAL_TIMEOUT=7200s` 下，即使当前实现也只能处理 ~240 页（VLM 单路径），而 v0.7 三级兜底在 7200s 内只能完成 **~38 页**。

**引擎调用次数统计：**

| 场景 | 引擎调用/页 | 验证调用/页 |
|------|------------|------------|
| Tier 1 首引擎通过 | 1 | 1 |
| Tier 1 两引擎均失败 | 2 | 2 |
| 三级全失败 | 4 | 4 |
| 三级都失败（并行 Tier 1） | 4 | 4 |

**→ 建议：** ① 明确编排主循环必须引入**页级超时**（单个引擎调用超时 + 单页全级总超时）；② Tier 3 本地 LLM 调用加入**超时熔断**（当前无单项超时配置）。③ 建议 TOTAL_TIMEOUT 默认值从 7200s 重新评估（参考 round4 评审结论）。

### 2.5 字形验证器性能开销

**各子模块开销分解：**

| 验证项 | 复杂度 | 实现方式 | 预估开销/页 |
|--------|--------|---------|------------|
| D4 字符数尖峰 | O(char_count) | 与基线比较 + 阈值判定 | <1ms |
| C1 跨页泄漏 | O(char_count) | 字符分布变化检测 | <2ms |
| 药材名/术语 KB 匹配 | O(char_count × avg_kb_lookup) | hash set / trie 查找 | 1–5ms |
| 形似混淆集 | O(char_count × avg_confusion_set) | hash set 查找 | <1ms |
| **合计** | O(char_count) | 纯内存操作 | **<10ms** |

**结论：不是性能瓶颈。**

- 每页 <10ms，即便最坏情况每页调用 4 次（三级全过）也仅 ~40ms/页
- 无 I/O（KB 已在 E3 说明「启动时一次性进程内镜像」——前提是**实现时**遵守此约束）
- 与引擎调用耗时（5–120s）相比低 3–4 个数量级

**残留风险：** 若 `term_kb` 知识库很大（>10 万条）且使用正则匹配而非哈希查找，可能膨胀到 50–100ms/次。建议在 E3 文档中明确匹配算法约束：**术语 KB 必须用前缀树或哈希集实现，禁止逐条正则匹配。**

### 2.6 Benchmark 数据更新频率与开销

**`registry.record()` 调用频率（伪代码）：**

每引擎每次结果 `registry.record(engine, success=True/false, glyph=verdict)`

| 场景 | 调用次数/页 | 200 页累计 |
|------|------------|-----------|
| Tier 1 一次通过 | 1 | 200 |
| 三级全走 | 4 | 800 |
| 最坏（三级全走 + 每级 2 引擎） | 6 | 1200 |

**写盘频率：** 若每次 `record()` 同步写盘，200 页书产生 200–1200 次写盘操作。

**性能风险取决于持久化策略：**

| 策略 | 每写入开销 | 200 次累计 | 800 次累计 | 风险 |
|------|-----------|-----------|-----------|------|
| JSON 全文重写 | O(current_size) | 200 × avg_size | 800 × avg_size | **中-高：文件越大越慢，O(n²)** |
| CSV 追加 | O(1) 追加 | ~毫秒级 | ~毫秒级 | 低 |
| SQLite 单行 INSERT | O(1) | ~毫秒级 | ~毫秒级 | 低 |
| 内存 buffer + 批量 flush | O(buffer_size) | 每 N 页 1 次写 | -- | 最低 |

**文档仅提及 `registry.record()`，未指定持久化格式和频率。存在 O(n²) 退化风险。**

**→ 建议：** 明确 benchmark 持久化策略为**追加式（CSV 追加 / SQLite INSERT / 结构化日志追加）**，禁止 JSON 全文覆写。同时支持**批量 flush**（每 10 页或书结束时写盘），避免单次成功/失败都触发 I/O。

### 2.7 内存使用峰值

**内存分解（200 页中医书）：**

| 组件 | 每实例 | 实例数 | 占用 | 备注 |
|------|-------|--------|------|------|
| PDF 页对象（Page 对象） | ~1KB | 200 | ~200KB | 仅句柄，轻量 |
| 渲染后的页图像（numpy） | ~6.5MB (150 DPI A4) | 1（逐页） | **~6.5MB** | 取决于 render_pages 是否流式 |
| 引擎中间结果（text） | ~1KB | 2（并行） | ~2KB | 文本结果很轻 |
| 字形验证结果 | ~0.1KB | 4（最坏） | ~0.4KB | 忽略 |
| Registry/Stats | ~0.5KB | 9 引擎 | ~5KB | 忽略 |
| pages_text 累积 | ~1KB | 200 | ~200KB | 文本累积 |
| BookPipeline 实例 | ~200–500MB | 1–2（并行） | **~200–1000MB** | 最大变量 |

**内存风险排序：**

1. **BookPipeline 实例（高）：** 每个实例加载 OCR 模型/词典 ≈ 200–500 MB。并行 2 个意味着 400–1000 MB。配合系统 6 GiB 可用内存，加上 OS 缓存和其他进程，**已接近上限**。
2. **PDF 渲染图像（中）：** 若 `render_pages` 非流式（一次性渲染所有），200 页 × 6.5MB = 1.3 GB。
3. **其他项（低）：** 合计 < 1 MB。

**→ 建议：**
1. v0.7 应禁止 Tier 1 并行（见 2.1 结论），从根源消除双 BookPipeline 内存叠加。
2. 明确 `render_pages` 必须实现为**生成器**（yield），禁止全量物化到列表。
3. 若保留并行选项，必须添加**内存预算检查**（预计峰值 > 可用内存 50% 时降级为串行）。

## 三、性能风险汇总

| 风险 | 级别 | 描述 | 触发条件 | 影响 |
|------|------|------|---------|------|
| R1 | **P0** | Tier 1 并行导致 CPU 争抢+吞吐下降 | 无 GPU 下 Tier 1 并发=2 | 同页完成时间 2×–4× |
| R2 | **P0** | 编排循环缺少 B6 双闸 | 大书（>50 页）或长尾页 | 资源耗尽/超限运行 |
| R3 | **P1** | 三级兜底最坏延迟 190s/页 | 全部引擎在慢路径上均失败 | 200 页书需 ~10.5h |
| R4 | **P1** | benchmark 持久化策略未定 | JSON 全文覆写实现 | O(n²) I/O 退化 |
| R5 | **P2** | BookPipeline 双实例内存叠加 | 允许并行时 | 可用内存可能不足 |
| R6 | **P2** | render_pages 非流式实现 | 全量渲染到列表 | 200 页 ~1.3GB |
| R7 | **P2** | 资源过滤含 TCP/CUDA 调用 | 每次 select_candidates 都探测 | 每书额外 120s+ |
| R8 | **P3** | 字形 KB 匹配用正则而非哈希 | term_kb 大词库 + 逐条正则 | 50–100ms/次验证 |

## 四、优化建议

### 必做项（P0，阻塞实施）

1. **Tier 1 并行默认关闭，仅 GPU 环境 opt-in**
   - `KZOCR_MAX_TIER1_CONCURRENCY` 默认值改为 1
   - 仅在探测到 GPU（`ProbeResult.has_gpu=True`）且显式配置 `KZOCR_MAX_TIER1_CONCURRENCY=2` 时允许并行
   - 并行时需验证可用内存 ≥ 1GB/实例 × 并发数 + 余量

2. **编排主循环恢复 B6 双闸**
   - `orchestrate_book` 的 `for page` 循环入口加入：
     - `page_num > config.max_pages → break`（书级截断）
     - `time.monotonic() - start > config.total_timeout → break`（每页时间闸）
   - 当前 `Budget` 对象仅用于调度器入口预算，无法替代循环级保护

3. **明确 benchmark 持久化策略**
   - 指定追加式写入（CSV 行追加 / SQLite INSERT）
   - 支持批量 flush（每 10 页或书结束时写盘）
   - 在文档中补充持久化策略说明（当前完全缺失）

### 建议项（P1–P2，强烈建议）

4. **引入页级超时 + Tier 3 超时熔断**
   - 单页整级处理的超时：max(Tier 1×N + Tier 2 + Tier 3 + 验证)
   - Tier 3（本地 LLM）单独设超时，超时后标记 UNCERTAIN 进入 HumanGate 而非无限等待
   - 建议默认值：单页超时 300s（5min），Tier 3 超时 180s

5. **资源过滤状态缓存**
   - VRAM/网络可达性改为状态位缓存，仅 probe 周期或错误事件触发时刷新
   - `select_candidates` 只读缓存，不做外部调用
   - probe 周期建议 30–60s（非每次选择都重探）

6. **render_pages 强制流式**
   - 生成器契约：`yield` 而非 `list` + `return`
   - 配合双闸：MAX_PAGES 截断在流式渲染的源头，而非渲染后丢弃

### 低优先（P3，可后续迭代）

7. **字形 KB 匹配算法约束**
   - 文档补充实现约束：术语 KB 用 `set`/`trie`，禁止逐条 `re.match`
   - 当前知识库规模（<5000 条）下哈希查找 <1ms，无需担心

8. **并行架构考虑 asyncio**
   - 当前伪代码用同步 `for engine in engines`，若将来需真并行，应使用 `asyncio.gather()` + `asyncio.wait(return_when=FIRST_COMPLETED)` 实现「先完成者获胜」，避免串行等待全部完成
   - 同步线程池也可接受，但要处理 GIL 问题

## 五、总结

### 架构方向评价

注册中心 + 调度器 + 字形验证器 + 编排循环的四层架构在抽象层面正确，将 v0.6 的硬编码 if-else 提升为动态路由，进步明确。

### 性能基线

| 指标 | 当前 v0.6（VLM-only） | v0.7 设计 | v0.7 修正后预期 |
|------|---------------------|-----------|---------------|
| 单页最优延迟 | ~30s | ~5s（Tier 1 直接通过） | ~5s |
| 单页最坏延迟 | 30s（无兜底） | ~190s | ~190s（但有超时保护） |
| 200 页完成时间 | ~1.7h | 最坏 ~10.5h | 受 TOTAL_TIMEOUT 限制 |
| 引擎调用/页 | 1 | 1–4 | 1–4 |
| 内存峰值 | ~500–800MB | ~700–1800MB | ~500–1000MB（修正并行后） |
| 双闸有效性 | 完整 | 缺失 | 恢复后生效 |

### 三把锁

v0.7 性能可行必须解决的三个核心矛盾：

1. **「并行」声言 vs CPU 争抢现实**——无 GPU 下 Tier 1 并行为反模式，必须默认串行、仅 GPU 环境 opt-in 并行
2. **「动态路由」vs 「双闸缺失」**——新架构失去了当前 `_run_vlm` 中每页都检查的 MAX_PAGES 和 TOTAL_TIMEOUT，必须恢复
3. **「三级兜底」vs 「预算契约」**——三级兜底带来 4×–6× 的延迟放大，需要页级超时和 Tier 3 熔断来约束

### 实施顺序

修正后建议重新评审以下方面是否就绪：
- B6 双闸在编排主循环中的正确实现
- benchmark 持久化策略的具体设计
- `render_pages` 的流式实现
