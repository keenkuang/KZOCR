# PM 评审报告：v0.7 自适应 OCR 引擎编排层

评审人：产品经理  
评审日期：2026-07-10  
评审对象：`docs/plans/ocr-engine-unification.v0.7.md`

---

## 一、总体判断

**有条件通过，但需大幅补强用户侧设计和体验验证。**

v0.7 的技术方向本身没有硬伤——把硬编码 if-else 抽成注册中心+调度器是架构正确的选择，工程团队需要这个抽象来降低后续新增引擎的维护成本。但方案目前呈现为纯技术设计文档，缺乏面向用户的价值论证和体验设计，部分决策存在将技术合理性等同于产品价值的风险。

**核心问题**：v0.7 解决的是"工程团队维护困难"的问题，而非"用户 OCR 效果不好"的问题。如果这是基础设施重构，应明确其定位；如果声称也改善用户体验，则需要用户侧证据。

## 二、用户价值评估

### 2.1 当前痛点诊断

当前（v0.6）的引擎选择机制：

```
KZOCR_USE_MOCK=1    → mock（桩数据，用于测试）
KZOCR_USE_VLM=1     → VLM 模式（SenseNova → PaddleOCR-VL 降级）
默认                   → kimi BookPipeline（失败则降级 mock）
KZOCR_REQUIRE_REAL=1 → 真实引擎失败直接抛错（调试用）
```

场景 × 用户感受矩阵：

| 用户场景 | 当前方式 | 用户感受 | 严重程度 |
|---------|---------|---------|---------|
| 日常批量 OCR | 设环境变量 → 跑 pipeline | 一次配置后基本不动 | 低痛 |
| 某引擎挂了一个 | 硬编码降级链兜底 | 用户无感知（可能降级到更慢/更差的引擎而不自知） | 中痛 |
| 想比较两个引擎结果 | 换环境变量重跑，手动对比 | 没有工具支持 | 中痛 |
| 新增一个引擎 | 改 run.py 的 if-else | 只有开发者会遇到 | 工程痛 |
| 知道某个引擎对古籍更准 | 无法指定，只能全局开关 | 用户缺乏控制权 | 中痛 |
| 某批 PDF OCR 特别慢 | 不知道哪个引擎是瓶颈 | 无法排查 | 低痛（有日志） |

**用户真实痛点优先级猜测**：
1. **结果不透明**（最高频）：用户不知道自己当前用的是哪个引擎，也不知道为什么这个引擎被选中——这是可感知的。
2. **无法按需指定**（中频）：用户有时知道某类书适合哪个引擎，但无法按书籍/批次指定。
3. **降级无声**（中频）：VLM 从 SenseNova 无声降级到本地 PaddleOCR-VL，用户只发现变慢了，不知道为什么。

### 2.2 v0.7 方案对用户痛点的覆盖

| 方案功能 | 解决的痛点 | 用户可感知的价值 |
|---------|-----------|---------------|
| 调度器自动选引擎 | 降级无声 → 可见 | 中等（需要用户可见调度决策） |
| 引擎注册中心 | 新增引擎维护 | 无（纯工程改进） |
| GlyphVerifier 字形验证 | 错字兜底 | 高（直接影响 OCR 质量） |
| 三级兜底 | 页级失败恢复 | 高（减少"这一页彻底废了"） |
| benchmark 数据 + kzocr benchmark | 性能不可见 | 中高（需要 UI 设计得好） |
| 并行引擎 | 速度慢 | 高（但实际收益取决于并行数配置） |

**最大价值点**：GlyphVerifier + 三级兜底。这是用户能直接感知到的质量提升——此前 OCR 错了就错了，没有校验环节。

**风险点**：调度器自动选择。如果用户发现"我想用的引擎没被选上"或"选了一个慢的"，反而会产生对系统的不信任感。

## 三、UX / 产品建议

### 3.1 必须让用户知道"谁干了什么"

当前方案完全没有提及用户如何感知引擎选择。建议：

**建议 A：在 pipeline 输出加"引擎报告页"**

每本书处理完成后，在日志末尾打印一段摘要：

```
=== 引擎报告 ===
书籍：金匮要略-卷上
总页数：48
引擎使用分布：
  Tier1 mineru     → 32 页 (PASS)
  Tier1 paddleocr  → 12 页 (PASS)
  Tier2 sensenova  → 4 页 (PASS)
字形验证通过率：46/48 (95.8%)
失败页：第 17, 23 页（已标记待人工校对）
总耗时：4m32s
==============
```

这让用户对"系统做了什么"有透明感。如果是静默自动选择而不告知，用户会不安。

**建议 B：提供 --engine-report <json> 参数**

导出结构化引擎使用报告，便于用户集成到自己的分析流程。

### 3.2 保留用户手动指定能力，比自动选择优先级更高

当前方案的调度器完全是自动的。从产品角度看，**用户的手动指定应覆盖自动调度**：

```bash
# 方案中已有：kzocr pipeline --engine sensenova     # 强制使用云端 VLM
# 方案中已有：kzocr pipeline --engine paddleocr       # 强制使用本地 OCR

# 建议补充：
kzocr pipeline --tier-order "1,3,2"                 # 自定义 tier 顺序
kzocr pipeline --prefer speed                       # "我要快的"（调度器在 PASS 引擎中选延迟最低的）
kzocr pipeline --prefer accuracy                     # "我要准的"（调度器基于 glyph_pass_rate 排序）
```

理由：
- 药房/医院用户对时效性敏感，可能愿意牺牲一点准确率换取速度。
- 研究人员可能相反，愿意多等几分钟换更高准确率。
- 自动调度应作为默认行为，但高级用户必须有覆写能力。

### 3.3 三级兜底的等待时间需要给用户预期

三级兜底意味着最坏情况下，一页 OCR 经过：
- Tier1（本地 OCR，1-2 个引擎并行，~5-10s）
- 字形验证（~0.1s）
- Tier2（云端 VLM，~10-30s）
- 字形验证（~0.1s）
- Tier3（本地 VLM/LLM，~10-30s）
- 字形验证（~0.1s）
- → 进入失败页标记

最坏情况一页可能需要 **30-70 秒**。100 页的书在最坏情况下（每页都失败前两个 tier）需要 **50 分钟到 2 小时**。

**建议 C：增加进度条和 ETA 估计**

当前 `_run_vlm` 每 5 页打一次日志，改进为：

```
[12:30:15] 正在处理第 23/48 页 | 用时 3m12s | 预计剩余 3m45s | 
Tier1: ████████░░ 80% PASS | Tier2: ██░░░░░░░░ 20% | ⏳ 0 页待人工
```

让用户对"还要等多久"有明确预期。

**建议 D：设置"单页最大耗时"硬上限**

除了全局 total_timeout，还应新增 `--max-time-per-page`（默认 120s），防止某一页卡死拖慢全书。

### 3.4 benchmark 命令的设计要点

方案提到新增 `kzocr benchmark`，但只说是"录入历史数据"。从产品角度，这个命令应该做：

```bash
kzocr benchmark status              # 查看当前所有引擎的健康状态
kzocr benchmark history --engine vlm  # 查看 vlm 的历史表现
kzocr benchmark run                  # 主动跑一次 benchmark（用测试集评估各引擎）
kzocr benchmark reset                # 重置历史数据（如换引擎版本后）
```

输出示例：

```
kzocr benchmark status
─────────────────────────────────────────────────────
引擎             状态        glyph 通过率    平均延迟   最后调用
─────────────────────────────────────────────────────
paddleocr        HEALTHY    96.2%          4.2s/页    10s 前
rapidocr         HEALTHY    94.8%          3.1s/页    10s 前
mineru           HEALTHY    97.1%          5.5s/页    10s 前
sensenova        HEALTHY    98.5%          12.3s/页   2m 前
paddleocr_vl16   DEGRADED   89.2%          18.7s/页   5m 前（1 次 408）
shizhengpt       UNAVAILABLE -             -          -
─────────────────────────────────────────────────────
推荐引擎（全局）：sensenova（准确率最高）或 rapidocr（最快）
```

### 3.5 与现有 CLI 的兼容性

方案中说 `run_engine()` 改为调用 Orchestrator，表面上不改变 `kzocr pipeline` 和 `kzocr smoke` 的命令行签名。但有两个需要考虑的影响：

**问题 1：mock 模式 vs 调度器**

当前 mock 模式在 `run_engine()` 入口处短路返回。v0.7 是将 mock 视为一个引擎注册到注册中心，还是保持短路？如果 mock 走注册中心，`kzocr smoke` 的确定性行为需要保证——mock 必须始终返回桩数据，不能因为"调度器认为 mock 的 glyph_pass_rate 低"而跳过它。

**建议 E**：mock 引擎保持特殊定位——`use_mock=True` 时跳过调度器直接返回，不参与任何自动选择逻辑。

**问题 2：KZOCR_REQUIRE_REAL 的语义变化**

当前 `REQUIRE_REAL=1` 意味着"真实引擎失败则抛错"。在多引擎调度模式下，"真实引擎失败"的语义变得模糊：
- 是指所有注册引擎都失败才抛？
- 还是 Tier1 全失败就抛？
- 还是用户指定的某个关键引擎失败就抛？

**建议 F**：重定义 `REQUIRE_REAL` 的语义。或者新增 `--fail-on-no-pass` 参数，语义为"没有任何引擎通过 glyph 验证时抛错"，更符合该参数的原始意图（debug 时不想吞错误）。

### 3.6 GlyphVerifier 的 UX 考虑

字形验证是 v0.7 最亮眼的功能，但用户需要理解 GlyphStatus 的含义：

| 状态 | 应如何向用户解释 |
|------|--------------|
| PASS | 可信，直接采用 |
| RARE | 罕见字，已采用但建议校对时留意 |
| UNKNOWN | 字形未在知识库中找到，默认放过但建议重点校对 |
| FAIL | 明确错误，已触发兜底 |
| UNCERTAIN | 字符数异常（可能漏字/多字），建议重点校对 |

在最终交付物中，应在页边/脚注标注可疑行。产出 markdown 中可嵌入不可见标记（如 `<!-- VERIFY: UNCERTAIN P17 L23 -->`），zai 校对台可据此高亮。

## 四、产品风险

### 4.1 自动调度可能降低用户信任（高风险）

如果用户发现同一本书两次跑结果不一致（因为调度器在不同时间选用了不同引擎），而没有清晰的报告说明差异来源，用户会认为系统"不稳定"、"不可靠"。

**缓解措施**：
- 每次运行输出完整的引擎选择决策树（类似前文建议的引擎报告）
- 提供 `--deterministic` 模式，固定引擎选择策略（例如总是选 glyph_pass_rate 最高的引擎，忽略延迟因素）
- 提供 `--pin-engine <name>` 强制指定引擎（应已有，但需确认）

### 4.2 三级兜底 + 字形验证的组合延迟风险（中风险）

每页 3 次引擎 + 3 次字形验证，按 100 页书籍估算：

| 场景 | 每页时间 | 总时间 | 用户感受 |
|-----|---------|-------|---------|
| 理想（Tier1 全部 PASS） | 5-10s | 8-17min | 可接受 |
| 中等（50% 需要 Tier2） | 平均 15s | 25min | 勉强可接受 |
| 最差（全部三级） | 40-70s | 67-117min | 不可接受 |

**建议**：添加 `--tier-limit N` 参数（默认 2），限制最大尝试 tier 数。同时设置"快速失败"策略：如果某页在 Tier1 连续失败 3 次（超过 `--tier-limit`），直接标记失败并记录，不浪费后续 tier 时间。

### 4.3 调度器"黑盒化"风险（中风险）

调度器基于 glyph_pass_rate × (1/avg_latency) 加权排序。但：
- glyph_pass_rate 是历史数据，对新书可能不代表实际表现
- 新注册引擎（无历史数据）如何初始化权重？全部设为 0 → 永远不会被选中？
- 某个引擎因网络波动临时延迟高 → 排名下降 → 后续更少被调用 → 排名进一步下降（冷启动陷阱）

**建议**：
- 新引擎初始时赋予"乐观"权重（所有引擎的平均 glyph_pass_rate，保守的平均延迟）
- 定期轮询调度（即使某引擎排名低，也以 5% 概率随机选择以更新其数据）
- 提供 `--engine-weight sensenova=2.0` 参数，允许用户人工调整权重

### 4.4 数据出境合规风险（低风险）

方案中 Tier2 包含云端视觉 LLM（如 siliconflow VLM），与现有 `allow_cloud_vision` 配置联动。需确认：
- 调度器自动选择 Tier2 引擎时，是否检查 allow_cloud_vision 标志？
- 若未设置该标志，自动调度是否仍会选择云端引擎？
- 文档中明确标识哪些引擎涉及数据出境

从方案看，资源过滤包含"当前状态 != UNAVAILABLE"，但未提及 `allow_cloud_vision` 约束。需补上。

### 4.5 从 v0.6 到 v0.7 的迁移体验（低风险）

当前用户通过环境变量控制引擎。v0.7 引入调度器后，用户可能会困惑于"我设了 KZOCR_USE_VLM=1，但调度器似乎没按我的意思选"。

**建议 G**：迁移期间保持向后兼容。如果用户设置了 `KZOCR_USE_VLM=1`，则跳过调度逻辑，沿用当前 VLM 直接模式。仅在未设任何旧式环境变量时启用调度器。过渡一个版本后废弃旧变量。

## 五、总结

### 5.1 必须做的修正

1. **加用户可见的引擎报告**（3.1 建议 A）——没有透明度的自动选择会消耗用户信任
2. **保留手动指定优先级高于调度器**（3.2）——自动是默认，但不能是唯一
3. **补充进度汇报**（3.3 建议 C）——长时间等待需要进度感
4. **mock 引擎不走调度器**（3.5 建议 E）——保证 `kzocr smoke` 确定性

### 5.2 强烈建议做的补充

5. `--prefer speed / --prefer accuracy` 偏好参数（3.2）
6. `--max-time-per-page` 单页耗时上限（3.3 建议 D）
7. `--tier-limit` 控制兜底深度（4.2）
8. 初始权重 + 轮询调度防止冷启动（4.3）
9. `allow_cloud_vision` 与调度器的联动检查（4.4）
10. 向后兼容旧式环境变量（4.5）

### 5.3 整体评价

v0.7 的技术架构合理且有前瞻性，但**目前处于"工程师写给工程师看"的阶段**。要把这个版本交付给用户，需要在以下三个方向补齐产品设计：

1. **透明度**：用户必须能知道引擎选择了什么、为什么、结果如何
2. **控制权**：自动调度是默认行为，但用户必须能以多种粒度覆写
3. **进度感**：延时增加（三级兜底）必须有相应的进度反馈

如果这三件事做不好，v0.7 反而会因为"变慢了还不告诉用户为什么"而降低 NPS。
